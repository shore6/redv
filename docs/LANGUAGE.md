# redv 言語仕様

`.rv` ファイルの文法と、シミュレーションエンジンの挙動(セマンティクス)を定義します。
概要・ビルド方法は [../README.md](../README.md) を参照してください。

---

## 1. 全体構造

- `logic name(input x, output y) { ... }` … レッドストーン回路の定義
- `module name() { ... }` … テストベンチ。`var` 宣言と `sim { ... }` ブロックを書く
- `param W = 4;` … トップレベルの整数定数。バス幅・sim 式から参照できる(§6.1)
- 文末は `;`。ブロックは `{}`。コメントは `//`(行)と `/* */`(範囲)
- ファイル内の全 `module` が **宣言順** に実行される(回路はモジュールごとに独立)

## 2. 宣言子

| 宣言 | 意味 |
|---|---|
| `input` / `output` | ポート。無条件で `reg` 型(= 点) |
| `wire` | **再利用可能な素子列**(端点を持たないテンプレート)。チェーンの中間に展開する |
| `reg` | 点。単体素子を代入する |
| `reg[N] a;` | **バス**。N 本の並列レーン `a[0]`..`a[N-1]`(§4.2)。各レーンは通常の点 |
| `const reg n = 15b;` | 素子も値も不変。**信号強度の指定が必須**(0–15、範囲外はエラー) |
| `mutable reg n = d;` | 素子は不変、値は回路によって変化 |
| `var` | テスト用の整数変数(C の `int` 相当)。**回路には組み込めない** |

> **名前の制約**: `reg` / `wire` / ポート(`input` / `output`)の名前に **素子名と衝突する名前** は
> 使えない(エラー)。具体的には「名前全体が素子列として解釈できる」もの全部 — 単体素子の
> `b` / `d` / `o` / `r` / `t`、コンパレータ `cc` / `cd`、`dx` / `r2` / `d3` のような付き素子、さらに `tb`
> (= `t` + `b`)のような連結も含む。これらはチェーン内で「名前付きの点」と「素子列」が区別
> できず曖昧になるため。`b2` / `cmp` / `in_b` のように素子列にならない名前を使う(`a` / `x` /
> `c` 単体は素子名ではないので可)。`var` は別名前空間でチェーンに現れないため制約しない。

## 3. 素子

| 表記 | 素子 | 動作 |
|---|---|---|
| `d` / `dx` | ダスト(点 / 十字) | 即時伝搬、1 個につき強度 −1 |
| `dn`(例 `d3`) | ダスト n 個 | `ddd...` と同じ |
| `r0` | 0tick リピータ | **遅延ゼロ**で再増幅(`in>0 ? 15 : 0`)。同一 tick で出力に伝わる(§3.3) |
| `r`, `r1`–`r4` | リピータ | n tick 遅延、出力は 15 に再増幅。1tick パルスも保持 |
| `t` | トーチ | 1 tick 遅延で反転(入力 > 0 → 0、入力 = 0 → 15) |
| `cc` / `cd` | コンパレータ(比較 / 減算) | 1 tick 遅延。サイド入力なしなら **強度をそのまま** 通す(`c` 単体はエラー) |
| `b` | ブロック | 即時、強度は 0 か 15 のみ |
| `o` | オブザーバ | 入力の **変化** を検出して 1 tick 遅れで 1tick・強度 15 のパルス(§3.4)。インラインチェーン専用 |

> `r` は `r1` と同じ(遅延 1)。`r5` 以上はエラー。`c` 単体はモード未指定のためエラー(`cc` / `cd` を使う)。

### 3.1 コンパレータのサイド入力

コンパレータは **後ろ(back)入力・横(side)入力・前(out)出力** の 3 端子を持つ。
出力は 1 tick 遅延で、モードにより次のように決まる:

- `cd`(減算): `out = max(0, back − side)`
- `cc`(比較): `out = back >= side ? back : 0`

**サイド入力なし(横未接続 = side 0)なら両モードとも `back` のパススルー** に退化する。
よってワイヤーチェーン内にインラインで置いた `cc` / `cd`(例 `a-d4ccd4-y`)は従来どおり強度中継として働く。

サイド入力を結線するには、コンパレータを **reg に代入して名前付きの点** にし、後ろ・横・前を
別々のワイヤーでつなぐ:

```rv
reg cmp = cd;        // 減算コンパレータ(cc = 比較)。修飾子は plain のみ・強度指定は不可
a - d3 - cmp;        // cmp が終端 → 後ろ(back)入力。減衰も普通に書ける
s - cmp.side;        // cmp.side → 横(side)入力
cmp - y;             // cmp が始端 → 前(out)出力
```

> **コンパレータ reg は宣言時初期化に限る。** 後ろ・横・前の 3 端子は宣言 `reg cmp = cd;` の
> 時点で確定する。`reg cmp; cmp = cd;` のような **後置代入は不可**(エラー)。

| 端子 | 書き方 | 役割 |
|---|---|---|
| back | `cmp`(チェーン終端) | 後ろ入力 |
| side | `cmp.side`(チェーン終端のみ) | 横入力 |
| out | `cmp`(チェーン始端) | 前出力 |

- back / side それぞれに複数ワイヤーが合流したら **強度の最大値**(ゲームの左右サイド max と一致)
- `.side` を **始端(信号源)** に使うのはエラー(横は入力専用)
- `.side` を **コンパレータ / リピーター以外の reg** に付けるのはエラー
- コンパレータ reg に `const` / `mutable` 修飾子・信号強度リテラルを付けるのはエラー

### 3.2 リピーターロック(リピーターのサイド入力)

リピーターも reg に代入して **名前付きの点** にすると、コンパレータと同じく
**後ろ(back)入力・横(side / ロック)入力・前(out)出力** の 3 端子を持つ。
横入力が `> 0` の間、出力は **ロック直前の値で凍結** され、後ろ入力が変化しても
追従しない。横入力が 0 に戻ると通常のリピーターに復帰する(ロックは 1 tick 反応)。

```rv
reg m = r;           // ロック可能リピーター(r / r1-r4。遅延は同じ規則)。修飾子は plain のみ・強度指定は不可
x - m;               // m が終端 → 後ろ(back)入力
lk - m.side;         // m.side → 横(ロック)入力
m - y;               // m が始端 → 前(out)出力
```

> **リピーター reg は宣言時初期化に限る。** コンパレータ reg と同じく 3 端子は宣言
> `reg m = r;` の時点で確定し、`reg m; m = r;` のような **後置代入は不可**(エラー)。

| 端子 | 書き方 | 役割 |
|---|---|---|
| back | `m`(チェーン終端) | 後ろ入力 |
| side | `m.side`(チェーン終端のみ) | 横(ロック)入力 |
| out | `m`(チェーン始端) | 前出力 |

- back / side の合流・`.side` の制約はコンパレータ(§3.1)と同じ
- リピーター reg に `const` / `mutable` 修飾子・信号強度リテラルを付けるのはエラー
- チェーン内インラインのリピーター(例 `a-r-b`)は従来どおりロックなし(横入力を持てない)
- **ゲームとの差異**: 本処理系は横入力が `> 0` なら **どんな経路(ダスト等)でもロック**
  する。ゲームでは横がリピーター/コンパレータからのときだけロックする点が異なる

### 3.3 0tick リピータ(`r0`)

`r0` は **遅延ゼロの組合せ増幅器**。通常のリピータ `r1`–`r4` が n tick 遅れて出力するのに対し、
`r0` は入力の変化を **同一 tick** のうちに出力へ伝える(`out = in > 0 ? 15 : 0`)。再増幅して
15 を出す点は普通のリピータと同じで、違いは遅延が無いことだけ。

```rv
x - d4 - r0 - d4 - y;   // 4 減衰 → 同tickで 15 に再増幅 → 4 減衰 = 11(遅延なし)
```

- **inline チェーン素子専用。** `reg m = r0;` のようにロック付き reg にはできない(ロックは前 tick の
  出力を凍結する仕組みだが、`r0` には保持する状態が無い)。`x - r0 - y;` の形で使う。
- **決定性**: `r0` は入力について単調(in が増えれば out も増える)なので、組合せ網の MAX 合流
  不動点にそのまま乗り、評価順に依存しない結果へ決定的に収束する。
- **ゲームとの差異**: ゲームに真の 0tick リピータは無い(遅延 1 が最小)。本処理系では「より高性能な
  回路をテキストで組む」ための拡張として用意している(issue #37)。0tick の **トーチ / コンパレータ**
  は反転・減算が非単調で、組合せループ検出を伴うエンジン拡張が要るため別 issue で扱う。

### 3.4 オブザーバ(`o`)

`o` は **入力の変化を検出して 1 tick パルスを出す** 順序素子。前 tick とサンプル値が異なれば
**強度 15 を 1 tick だけ** 出力し、変化が無ければ 0。立ち上がり・立ち下がり・強度変化(例 5 → 10)の
**すべてを「変化」として拾う**(変化全部モード)。

```rv
x - o - y;   // x が変化した次の tick に y が 1tick だけ 15 になる
```

意味論は `out(T) = in(T-2) != in(T-1) ? 15 : 0`。入力が回路へ反映される 1 tick と、隣接 2 サンプルの
比較による 1 tick で、**入力変化の 1 tick 後に 1 tick 幅のパルス** が出る(履歴 2 段に乗る順序素子)。

- **inline チェーン素子専用。** トーチと同じく `reg p = o;` のように reg には置けない(横端子を
  持たず、reg 化しても端子が増えないため)。`x - o - y;` の形で使う。
- 入力が定常(変化なし)なら出力は 0 に整定するので、**`#init` で定常状態になる**(発振しない)。
- **ゲームとの差異**: ゲームのオブザーバはブロック状態の更新で発火し、向き・QC(疑似配線)・
  出力面などの空間的性質を持つが、本処理系は素子レベルで **入力信号の変化のみ** を見る(向き・QC は
  扱わない)。立ち上がり / 立ち下がりだけを拾う **エッジ亜種** は issue #58 で扱う。

## 4. ワイヤーと点

2 つの reg / ポートを素子列でつなぐには、**チェーン文**(`from -素子...- to ;`)を直接書く。
wire 変数の宣言は不要。

```rv
x-ddr2brdccbr4d3-y;   // x と y を素子列でつなぐ(チャンクは連結して書ける)
x-d-r2-t-y;           // '-' で区切って書いてもよい
x-y;                  // 長さ0のチェーン(直結)
```

- チェーンの先頭と末尾は **宣言済みの reg / ポート**、中間は **素子 or wire 名**
- 1 つの reg に複数のチェーンが合流した場合、**強度の最大値** をとる(ゲーム仕様準拠)
- 端点に `.side` を付けるとコンパレータ / リピーター reg の横入力に結線(§3.1 / §3.2)。中間チャンクには付けられない

### 4.0 wire(再利用可能な素子列)

`wire` は **端点を持たない素子列のテンプレート** で、チェーンの中間チャンクとして展開できる。
同じ wire を複数箇所で使うと、**各箇所で独立した素子列**として展開される(物理的に別物)。

```rv
wire seg;
seg = d2ccd2;             // ダスト2 + コンパレータ + ダスト2 の素子列を定義
x-seg-m;                  // x を seg 経由で m へ
y-seg-m;                  // 同じ seg を y -> m にも再利用(各々別インスタンス)
```

- wire の代入(`w = ...`)は **接続ではなく素子列の定義**。端点(reg/ポート名)や `.side` は書けない
- 単一素子なら `w = r4;` のように書ける。複数素子は `w = d4-cc-d4;` も `w = d4ccd4;` も可
- wire は他の wire を含められる(`wrap = base-r-base;`)。**循環参照はエラー**
- 未定義(宣言のみで未代入)の wire をチェーンで使うのはエラー
- wire の再代入は **警告** を出し、最後の代入が採用される
- wire 自体はチェーンの **端点にはできない**(端点は reg / ポートのみ)

### 4.1 logic の階層化(インスタンス化)

`logic` の本体から別の `logic` を呼び出して入れ子の回路を組める。

```rv
logic and2(input x1, input x2, output y) {
    reg na, nb, o;
    na = not_gate(x1);      // out = callee(args...)
    nb = not_gate(x2);
    o  = or_gate(na, nb);
    y  = not_gate(o);       // a & b = !( !a | !b )
}
```

- 形は `out = callee(args...);`。`out` は **このlogic内の reg / 出力ポート**、`args` は
  **reg / 入力ポート名**(wire や素子・数値リテラルは不可)
- `out` には下位 logic の **出力ポートが結線**(減衰なし直結)され、`args` の各ノードが下位の
  **入力ポートを駆動** する。下位インスタンスは sim 変数とは無関係に回路として展開される
- 束縛は **出力ポートちょうど 1 個**。ただしその 1 ポートは **バス**(`output[N]`)でよく、
  バス出力 = **同種の多出力**(N レーン)になる(§4.3)。異種多出力のタプル束縛は未対応
- 引数の個数は下位 logic の入力ポート数と一致する必要がある(各引数は reg/ポート、またはバス全体)
- **再帰インスタンス化**(自己・相互を含む循環呼び出し)は **エラー**

### 4.2 バス(`reg[N]` — 複数レーンの束)

`reg[N] name;` は **N 本の並列レーン**(バス)を宣言する。各レーン `name[0]` .. `name[N-1]` は
通常のスカラ点で、バスは「N 本のスカラ点・スカラ操作」への **純粋な糖衣**。エラボレーション時に
N 本のスカラチェーンへ展開されるだけなので、シミュレーション意味論(合流 max・順序素子・`#init`)は
スカラ点と完全に同一(`circuit` エンジンはバスを一切知らない)。

```rv
reg[4] in;            // バス宣言: in[0]..in[3](`[N]` は reg と名前の間)
reg[4] buf;
a - in[0];            // レーンを添字 [k] で個別に指定(スカラ点として使える)
in - r - buf;         // ★ バス全体: 1 行が 4 レーンぶん(各レーンに独立したリピータ)に展開
buf[0] - y;           // レーンを取り出して出力ポートへ
```

| 書き方 | 意味 |
|---|---|
| `reg[N] a;` | 幅 N のバス宣言。`[N]` は `reg` と名前の間に置き、宣言列(`reg[N] a, b;`)全体に適用。`N` はリテラルのほか param 定数・定数式(`reg[W]` / `reg[W+1]`、§6.1)も可 |
| `a[k]`(k は整数リテラル, 0 ≤ k < N) | レーン k。スカラ点とまったく同じに使える |
| `a`(チェーン端点・添字なし) | バス全体。同幅の相手と element-wise に接続 |
| `A - 素子列 - B;`(両端バス) | 同幅なら **レーンごとに独立した素子列** で接続(中間 wire 名も各レーン独立展開) |

- **幅整合は厳格**: バスチェーンの両端は **同じ幅** でなければならない。スカラとバスの混在
  (幅不一致)はエラー。1 レーンだけ繋ぐときは添字 `name[k]`(= スカラ)を使う。
- バス名(ベース名)も §2 の **名前の制約**(素子名と衝突する名前は不可)を受ける。
- バスは **チェーンの端点専用**。中間チャンクには置けない(`-` の左右が端点)。

### 4.3 バスポート(`input[N]` / `output[N]`)

ポートもバスにできる。`input[N] a` / `output[N] b` は N レーンの並列ポートで、本体では
内部バス reg(§4.2)と同じく添字 `a[k]` / バス全体 `a` で使える。

```rv
logic and4(input[4] a, input[4] b2, output[4] y) {
    reg[4] na; reg[4] nb; reg[4] o;
    a - t - na;        // na = NOT a(4 レーン一括)
    b2 - t - nb;
    na - o; nb - o;    // o = na OR nb(各レーンで max 合流)
    o - t - y;         // y = NOT o = a AND b2(ビット単位 AND)
}
```

- バス出力ポートは **同種の多出力**(N レーン)。`out = g(args...)` の `out` がバス var なら
  レーン対応で束縛される(§5)。階層インスタンスでも親のバス reg/ポートと同幅なら結線できる。
- 引数 / 出力先とポートは **形(スカラ / バス)と幅が一致** しなければならない(不一致はエラー)。

### 4.4 スライス `a[hi:lo]` と連結 `{a, b}`

チェーンの **端点**(`-` の左右)には、レーンの **部分範囲(スライス)** と複数端点の
**連結** を書ける。どちらもバス糖衣の範囲で完結し、**レーン列へ展開するだけ**(`circuit`
エンジンは無変更)。端点はレーン列に解決され、両端の幅(レーン数)が一致すれば
element-wise に接続される。

```rv
a[3:0] - r - b[3:0];      // 部分バス(レーン範囲)を素子列でつなぐ
a[3:0] - b[0:3];          // 向きを変えるとビット反転配線(a[3]->b[0], ..)
hi - {carry, sum};        // 連結を一括配線(幅は各要素の和)
{a[2:0], a[3]} - out[3:0];// 連結 + スライスで 1 ビット左ローテート
```

| 書き方 | 意味 |
|---|---|
| `a[hi:lo]` | スライス(**包含**)。`hi >= lo` で降順 `[hi, hi-1, .., lo]`、`hi < lo` で昇順 `[hi, .., lo]`。`a[k:k]` は `a[k]` と等価(幅 1)。`hi`/`lo` は範囲内のリテラル |
| `{e1, e2, ...}` | 連結。各要素 `ei` は スカラ点 / レーン `a[k]` / スライス `a[hi:lo]` / バス全体 のいずれか。**左から順にレーンを連接**し、幅は各要素の和 |

- **バス全体名はインデックス昇順** `[0..N)` でレーンを供給する(既存のバス全体チェーンと同じ)。
  ビット順を制御したいときはスライス(`a[hi:lo]`)で明示する。
- スライス / 連結は **端点専用**。中間チャンク(素子列)・`=` の右辺には置けない。連結要素に
  `.side` やネスト `{{..}}` は不可。
- 幅整合は通常のバスチェーンと同じく **厳格**。範囲外添字・幅不一致・非バスのスライスはエラー。

> **未対応(将来拡張)**: バスの **plain 以外**(`const` / `mutable`・初期化子・コンパレータ/
> リピーターのバス)、バスを **代入の左右**(`a = ...`)や **wire 素子列** に置くこと、`wire[N]`、
> 異種多出力のタプル束縛、**バスレーンを logic 引数に直接渡すこと**(`g(x[0])`。バス全体を渡すか、
> 一旦スカラ var へ写す)。スライス添字は当面 **リテラルのみ**(`param` での式化は将来)。

## 5. sim ブロック

```rv
module test() {
    var x, y;
    sim {
        x = 0;
        y = not_gate(x);   // 初回で回路をインスタンス化、変数を入出力に束縛
        #init              // 定常状態まで待つ($time = 0 となる)
        x = 10;
        #1                 // 1 tick 進める
        ?monitor("時刻=% | x=% | y=%\n", $time, x, y);
    }
}
```

- `v = logic名(変数...)` … 同じ「logic 名 + 引数列」は同一インスタンス。入力変数は継続的に束縛され、
  出力変数は毎 tick 更新される。**バスポート**(§4.3)には **バス var を丸ごと** 渡す/受ける:
  `var[N] x, y; y = g(x);` はレーン対応で束縛される(スカラポートにはスカラ var、形と幅が一致必須)
- `var[N] x;` … **バス var**(N 本のレーン `x[0]`..`x[N-1]`)。`x[k]` でレーンを読み書きする
  (添字は実行時に評価する整数式。`for` ループで `x[i]` も可)。バス var を **添字なしで** スカラ式に
  使うとエラー(レーン指定が必要)。`x = 式;` は **全レーンへブロードキャスト**(例 `x = 0;` で一括クリア)
- `v = 式;` … var への代入。`v = 式 ~ w;` と書くと **パルス代入**: `v` に値を入れた後、
  `w` tick 経過すると **自動的に 0 に戻す**(代入 → ウェイト → 0 代入 を手で書かずに済む)。
  `w` は 1 以上の整数(式可、評価は代入時)。`w` tick は **実行されたあらゆる tick**(`#n` /
  `wait(n)` / `#init`)を数える。期限到来前に同じ var を通常代入すると保留中パルスは解除される(§7.6)
- `#n` … n tick 進める。`#init` … 更新キューが空(定常状態)になるまで待つ
- `#until(cond)` … `cond` が真(!= 0)になるまで tick を進める **イベント駆動待機**。出力が立つまでの
  tick 数を知らなくても「立つまで」待てる(非同期回路向け)。`$time` は `#n` 同様に進む(完了時の
  `$time` が経過 tick になる)。**条件が最初から真なら 0 tick**(`while` 同様、進入時に評価)。
  `INIT_TIMEOUT`(§6)を超えてもなお偽ならエラー(発振 or 永遠に成立しない条件)。`cond` は monitor 等と
  同じ sim 式(比較・論理演算可)。`examples/until_wait.rv`
- `?monitor(...)` … Verilog の `$monitor` 風。sim 開始時に登録され、**すべてのウェイト
  (`#init` / `#n` / `#until`)完了直後** に自動実行される(sim 内のどこに書いても全時刻が出力される)
- `monitor(fmt, ...)` … その場で出力。`%` が次の引数を 10 進で消費、`%N`(例 `%2`)で **最小幅**(右寄せ
  スペース詰め)を指定、`%%` で `%` リテラル。**隣接プレースホルダ**は `%%` がエスケープと衝突するため
  幅付き(例 `%1%1%1%1`)で書く。型サフィックス `%d` / `%t` は廃止(エラー。issue #17)
- `assert(cond)` … `cond` が偽(= 0)なら **失敗** を stderr に記録する(行番号 + 式を表示)。
  `expect(actual, expected)` … 不一致なら「実際の値 / 期待値」を stderr に記録する。**いずれも
  失敗しても sim は止まらず継続**し、全 module の sim 実行後に `N assertion(s), all passed`、
  または失敗があれば `assertions: M of N failed` を出して **非ゼロ終了** する。`monitor`(stdout)に
  頼らず **終了コードで合否が分かる** 自己検証テストベンチを書ける(`examples/assert_selfcheck.rv`)
- `wait(n)` … `$time` を進めず、monitor も発火させずに n tick 待つ(発振回路用)
- `clock(var, N)` … **テスト用クロック生成シュガー**。スカラ var を「各レベル **N tick 保持**」で
  `0` / `15` に自動トグルする(full period = 2N、50% デューティ)。呼び出し直後は **Low(0)**、以降
  N tick ごとに反転。`clock()` 自体は時刻を進めず、**後続の `#n` / `wait` / `#init` / `#until` が
  tick を刻む間**にトグルする(パルス代入と同型の遅延機構)。同じ var への通常代入でクロックは解除される。
  `N` は 1 以上の整数(式可)。`var` は宣言済みのスカラ var(バス var 不可)。monitor で読む var の値は
  **次 tick に適用される値**を映すため 1 tick 先行して見える(パルス代入 `v = x ~ 1` と同じ挙動)。
  デューティ比・初期位相の指定は将来拡張(issue 参照)。`examples/clock_sugar.rv`
- `v = scan()` … **stdin** から空白/改行区切りの整数を 1 つ読み、変数 `v` に代入する。
  EOF(入力切れ)・非数値は **エラー**。回路へ束縛した変数は読み取り後も 0–15 にクランプされる。
  テストでは `redv foo.rv < input.txt` のように入力を固定すれば決定的に再現できる
- `$time` … `#init` 完了時(未使用なら sim 開始時)を 0 とした経過 tick
- `if` / `while` / `for` … C 準拠(`for (i = 0; i < 4; i = i + 1) { ... }`)

#define BURNOUT_WINDOW 30     // 同: 監視窓(既定 30 tick)

### 6.1 param(パラメータ定数)

`param NAME = <定数式>;` でトップレベルに **整数定数** を宣言する。バス幅や sim 式から
名前で参照でき、幅をリテラル固定せず **1 定義を複数幅で再利用** できる(issue #41)。

```rv
param W  = 4;        // 定数
param W2 = W + 1;    // 既出の param を使った定数式も可(前方参照は不可)

logic notN(input[W] a, output[W] y) { a - t - y; }   // 幅を param で

module m() {
    var[W] x;        // バス幅に param
    var[W2] y;       // 定数式の結果(= 5)幅
    var i;
    sim {
        x = 0;
        for (i = 0; i < W; i = i + 1) { x[i] = 15; }   // sim 式でも参照可
    }
}
```

- **値の解決はパース時**。`param` / 数値 `#define` の値を表に持ち、`[ ... ]` 幅と sim 式の両方が
  この表を引く(数値 `#define W 4` もそのまま幅・式に使える)。
- **幅 `[expr]` は定数式**を受理する: リテラル・param・`+ - * / %`・単項 `-`・括弧。
  `reg[W+1]` / `var[2*W]` など。評価結果は **1 以上**(0 以下はエラー)。
- 定数式に使えないもの: `$time`、バス添字(`W[0]`)、比較・論理演算子。未定義名はエラー。
- sim 式では **同名の `var` が優先**(param を隠す)。`var` に無い名前のときだけ param を引く。
- ゲームとの差異・将来拡張: logic ごとのジェネリック幅 `logic adder #(W=4)(...)` は **Phase 2**(未実装)。

## 7. シミュレーション・セマンティクス(設計判断の確定値)

1. **ティック粒度** — `#1` = 1 レッドストーンティック。ゲームティック素子・1rstick 未満のパルスは
   扱わない。
2. **同一ティック内の評価順序** — 各 tick で「順序素子の出力確定 → 組合せ網(ダスト / ブロック)の
   不動点解決 → 順序素子の入力サンプリング」の 3 相。合流は max をとる単調更新なので不動点は決定的に
   収束し、走査順に結果が依存しない(逐次キュー方式と同じ結果を、順序非依存で保証する実装)。
3. **信号伝搬** — 減衰はティックごとに動的計算。ダスト 1 個につき −1、合流点は最大値。
4. **素子の更新タイミング**
   - ダスト / ブロック: 即時
   - 0tick リピータ `r0`: `out(T) = in(T) > 0 ? 15 : 0`(同一 tick・組合せ評価。§3.3)
   - リピータ `rn`: `out(T) = in(T−n) > 0 ? 15 : 0`(1tick パルス保持)。
     ロック付き(横入力あり)は横入力 `> 0` の間 `out(T) = out(T−1)`(凍結)に退化する
   - トーチ: `out(T) = in(T−1) > 0 ? 0 : 15`
   - コンパレータ: `out(T) = f(back(T−1), side(T−1))`。サイド未接続は side = 0 なので
     `out(T) = back(T−1)` のパススルーに退化(`cd`: `max(0, back−side)` / `cc`: `back>=side ? back : 0`)
   - オブザーバ: `out(T) = in(T−2) != in(T−1) ? 15 : 0`(隣接 2 サンプルの変化検出 = 1tick パルス)
5. **#init の定常判定** — 全ノード値が不変かつ全順序素子のパイプラインが一様になったら終了。
   `INIT_TIMEOUT` tick を超えるとエラー(発振回路向けに `wait(n)` を用意)。`#until(cond)` も
   同じ `INIT_TIMEOUT` を流用し、`cond` が真になるまで tick を進める(超過でエラー)。
6. **入力変更の反映** — `a = 10;` は次のティックの先頭で回路に反映(Verilog のノンブロッキング相当)。
   変数は `logic(引数)` で束縛したものだけが回路に影響する。**パルス代入** `a = v ~ w;` は
   `a` に `v` を代入したうえで「`w` tick 後に `a = 0`」を予約する。減算は各 tick 末(出力反映後)に
   行うので、`v` は **その後ちょうど `w` tick** 入力に適用されてから 0 に戻る。`w < 1` はエラー。
   期限前に `a` を通常代入すると予約は解除される(パルス代入なら新しい幅で置き換わる)。
7. **観測タイミング** — `monitor` / 出力変数は **ティック処理完了後** の値を見る。
8. **同一 reg への複数チェーン合流** — ゲーム仕様通り最大値。
9. **エラー / 警告(厳しめ)**
   - 範囲外の var → input: 0/15 に丸めて **警告**(変数ごとに 1 回)
   - const reg の範囲外強度・再代入: **エラー**
   - 未接続の output ポート: インスタンス化時に **エラー**
   - wire / reg 素子の再代入: **警告**(最終代入を採用)
   - トーチ焼き切れ: 窓内トグル超過で **警告** + 一定期間強制 OFF
   - 名前重複・未宣言変数・引数個数不一致・`c` 単体・`r5` 以上: **エラー**
   - reg / wire / ポート名が **素子名と衝突**(名前全体が素子列として解釈できる: `b` / `r` /
     `cd` / `tb` 等): **エラー**(§2 の名前の制約)
   - リピーター / コンパレータ reg を **宣言後に代入**(`reg m; m = r;`): **エラー**
     (宣言時初期化 `reg m = r;` に限る。§3.1 / §3.2)。トーチ `t` は reg に置けない
   - **診断表示**: エラー / 警告は Rust 風の **キャレット表示**(`--> file:line:col` + ソース行 +
     `^` 下線)で stderr に出す。構文(字句 / 構文解析)エラーは **正確な列** を、エラボレーション
     エラーは **行レベル**(行内容を下線)を指す(issue #47)
