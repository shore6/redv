## 1. 概要

### 1.1 全体構造

`.rv` ファイルは次の 4 種類のトップレベル要素で構成する。

- **`logic name(input x, output y) { ... }`**:レッドストーン回路の定義
- **`module name { ... }`**:テストベンチ。`var` 宣言と `sim { ... }` ブロックを書く
- **`param NAME = <定数式>;`**:トップレベルの整数定数(§8.3)
- **`#define` / `#include`** などのディレクティブ(§8.1, §8.2)

ファイル内の全 `module` は宣言順に実行される。
回路はモジュールごとに独立であり、ある module で組んだ回路を別の module から見ることはできない。

module は logic と違って引数を取らないため、名前の直後に `()` は書けない
(旧記法 `module name() { ... }` は新記法へ誘導するエラーになる)。

文末は `;`、ブロックは `{}`、コメントは `//`(行)と `/* */`(範囲)を使う。

### 1.2 最小例

トーチ 1 本で NOT を作り、入力を駆動して観測するまでの最小例を示す。

```rv
logic NOT(input x, output y) {
    x-t-y;                       // x と y をトーチでつなぐ
}

module test {
    var x, y;
    sim {
        x = 0;
        y = NOT(x);         // インスタンス化して変数を入出力に束縛
        #init                    // 定常状態まで待つ($time = 0)
        x = 10;  #1
        ?monitor("t=% x=% y=%\n", $time, x, y);
    }
}
```

`-` で素子列をつなぐ書き方(チェーン文)が回路記述の基本である(§5.1)。
`sim` ブロックの記法は §7 で扱う。

### 1.3 数値リテラル

整数リテラルは 10 進、2 進、16 進の 3 形式で書ける。

| 書き方 | 例 | 値 |
|---|---|---|
| 10 進 | `10`, `255` | そのまま |
| 2 進 | `0b1010`, `0b1111` | 10, 15 |
| 16 進 | `0xf`, `0xff`, `0x10` | 15, 255, 16 |

3 形式とも字句解析で同じ整数トークンに落ちる。
強度、バス幅、`param`、`#define`、sim 式、tick 数など、整数を受け付ける位置すべてで等価に使える。

接頭辞 `0b` / `0x` の直後に有効な数字が無いとき(`0b` 単独、`0x` 単独、または `0xg` のように非数字が続くとき)は誤記としてエラーで停止する。
2 進リテラル直後に `2`–`9` の十進数字が続く記述(`0b12` など)も typo 判定で停止する。

## 2. 名前と宣言子

回路に登場する「点」と、テストベンチで使う変数を宣言する構文を扱う。

### 2.1 宣言子の一覧

各宣言子の意味は次の表のとおり。
詳細は §3(修飾子と初期化)、§4(素子の表記)、§6(バス)を参照。

| 宣言子 | 意味 |
|---|---|
| `input` / `output` | ポート。点として扱う(`reg` と同じ性質) |
| `wire` | 再利用可能な素子列(端点を持たないテンプレート、§5.2) |
| `reg` | 点。単体素子を代入できる(§3) |
| `reg[N] a;` | バス(N 本の並列レーン、§6) |
| `const reg n = 15;` | 固定強度の定数(0–15 の数値で初期化が必須、§3) |
| `mutable reg n = d;` | 素子は不変、値は回路によって変化(§3) |
| `var` | テスト用の整数変数(`sim` 内のみで使う、回路へは組み込めない) |
| `param NAME = <定数式>;` | トップレベルの整数定数(§8.3) |

### 2.2 名前の制約

`reg` / `wire` / ポート(`input` / `output`)の名前は、素子名と衝突してはならない。
具体的には、名前全体が素子列として解釈できる文字列を禁止する。

衝突する例は次のとおり。

- 単体素子の `d` / `o` / `r` / `t`
- コンパレータ `cc` / `cd`
- オブザーバのエッジ亜種 `op` / `on` / `oe`
- 遅延つきの `r2` / `d3` のような表記
- これら素子の連結 `td`(= トーチ + ダスト) など

衝突する名前を許すと、チェーン内で「名前付きの点」と「素子列」が区別できない。
`b2` / `cmp` / `in_b` のように、素子列としては解釈できない名前を使う。
`a` / `b` / `x` / `c` 単体は素子名ではないので使ってよい。

なお `var` はチェーンに現れない別名前空間なので、この制約を受けない。

## 3. 修飾子と初期化

`reg` の宣言には修飾子と初期化子を付けられる。
3 通りの組合せがあり、それぞれ「素子と値」の固定度が異なる。

### 3.1 plain(無修飾)

```rv
reg a;                 // 点だけを宣言、何の素子もまだ決まっていない
reg c = r2;            // 単体素子(遅延 2 のリピータ)を素子として代入
reg m = r;             // ロック可能リピーターの「名前付きの点」(§5.3)
reg cmp = cd;          // 減算コンパレータの「名前付きの点」(§5.3)
```

素子と値の両方がチェーン文(§5.1)によって決まる、最も自由度の高い宣言である。

### 3.2 const

```rv
const reg n = 15;      // 強度 15 で固定(0b1111 / 0xf と書いても同じ)
const reg z = 3;       // 強度 3 で固定
```

`const reg` は値が不変の固定強度定数である。
0 から 15 の数値による初期化が必須で、範囲外はエラーとなる(§10)。
数値に素子トークンを添える旧記法(`15b` / `3d`)は、ブロック素子とあわせて廃止した(issue #75)。

### 3.3 mutable

```rv
mutable reg n = d;     // 素子はダスト固定、値は回路の駆動で変化する
```

`mutable reg` は素子を確定させたまま、値だけは回路の駆動で変化させる宣言である。

### 3.4 コンパレータ reg とリピーター reg の制約

`reg cmp = cd;` / `reg m = r;` のように、コンパレータやロック可能リピーターを宣言時に代入する形は「名前付きの点」を作る特殊な構文である(§5.3)。
この形には次の制約がある。

- 修飾子は plain のみ(`const` / `mutable` は不可)
- 信号強度リテラルを付けるのは不可
- 宣言時初期化に限る(`reg cmp; cmp = cd;` の後置代入は不可)

宣言時に代入する形に限定するのは、後ろ・横・前の 3 端子(§5.3)を宣言と同時に確定させるためである。

バス宣言 `reg[W] m = r;` も同じ制約で書ける(§6.5)。
この形は W レーンぶんの 3 端子をまとめて宣言する。

## 4. 素子

回路を組み立てる最小単位である素子を一覧で示す。
各素子はチェーン文(§5.1)の中に並べて使う。
ゲームのレッドストーン素子との挙動の違いは §11 にまとめてある。

### 4.1 一覧表

| 表記 | 素子 | 振る舞い | 詳細節 |
|---|---|---|---|
| `d` | ダスト | 即時伝搬、1 個につき強度 −1 | §4.2 |
| `dn`(例 `d3`) | ダスト n 個 | `ddd...` と同じ | §4.2 |
| `r`, `r1`–`r4` | リピータ | n tick 遅延、出力は 15 に再増幅 | §4.3 |
| `r0` | 0tick リピータ | 遅延ゼロで再増幅、`out = in > 0 ? 15 : 0` | §4.3 |
| `t` | トーチ | 1 tick 遅延で反転 | §4.4 |
| `cc` / `cd` | コンパレータ(比較 / 減算) | 1 tick 遅延、サイド入力で振る舞いが変わる | §4.5 |
| `o` | オブザーバ | 入力の変化を検出して 1 tick パルス | §4.6 |
| `op` / `on` / `oe` | オブザーバ(エッジ亜種) | 立ち上がり / 立ち下がり / 2値エッジのみ検出 | §4.6.1 |

`r` は `r1` と同じ(遅延 1)である。
`r5` 以上はエラー、`c` 単体(モード未指定)もエラーとなる。

### 4.2 ダスト(`d`)

ダストは即時伝搬する減衰素子である。
1 個につき強度を 1 下げ、合流点では最大値をとる。

`d3` のような表記は `ddd` と等価である。
ゲームでは配置によって「点 / 十字」など形状が変わるが、本処理系では形状を持たない(§11.1)。

### 4.3 リピータ(`r`, `r1`–`r4`, `r0`)

リピータは入力を 15 に再増幅する素子である。
チェーン文の中にインラインで置く形と、`reg m = r;` で「名前付きの点」にしてロック入力を許す形(§5.3)の 2 種類の使い方がある。

通常のリピータ `r1`–`r4` は n tick 遅延で出力を更新する。
1 tick パルスも保持できる(つまり「次の tick で消える入力」も出力に乗る)。

`r0` は遅延ゼロの組合せ増幅器である。
`r1`–`r4` が n tick 遅れて出力するのに対し、`r0` は入力の変化を同一 tick のうちに出力へ伝える。
出力規則は `out = in > 0 ? 15 : 0` で、再増幅して 15 を出す点は普通のリピータと同じ。

```rv
x - d4 - r0 - d4 - y;          // 4 減衰 → 同 tick で 15 に再増幅 → 4 減衰 = 11
```

`r0` には次の制約がある。

- インラインチェーン素子専用で、`reg m = r0;` のようにロック可能 reg にはできない
- ロックは前 tick の出力を凍結する仕組みだが、`r0` には保持する状態がないため

`r0` は入力について単調な素子なので、組合せ網の MAX 合流不動点にそのまま乗り、評価順に依存しない結果へ決定的に収束する。
0tick のトーチやコンパレータは非単調なので別扱いとなる(§11.4)。なお現在は未実装である。

### 4.4 トーチ(`t`)

トーチは入力を反転する 1 tick 遅延の素子である。

- 入力が `> 0` のとき出力は 0
- 入力が `= 0` のとき出力は 15

短時間に何度もトグルすると焼き切れ警告が出て、一定期間強制 OFF になる(§8.1 の `BURNOUT_*`、§10)。

### 4.5 コンパレータ(`cc`, `cd`)

コンパレータは 1 tick 遅延の素子で、モードが 2 種類ある。

- `cc`(比較):`out = back >= side ? back : 0`
- `cd`(減算):`out = max(0, back − side)`

サイド入力(横入力)を結線していない場合、`side = 0` として扱う。
このとき両モードとも `back` のパススルー(`out = back`)に退化するので、チェーンの中継素子として安全に使える。

```rv
x - d4 - cc - d4 - y;          // インラインの cc および cd は中継として働く(side = 0)
```

サイド入力を結ぶには「名前付きの点」にする(§5.3)。

### 4.6 オブザーバ(`o`)

オブザーバは入力の変化を検出して 1 tick パルスを出す順序素子である。
前 tick とサンプル値が異なれば強度 15 を 1 tick だけ出力し、変化がなければ 0 を保つ。

ゲームの仕様と同様に、立ち上がり、立ち下がり、強度変化(例 5 → 10)のすべてを「変化」として拾う。

```rv
x - o - y;                     // x が変化した次の tick に y が 1 tick だけ 15 になる
```

意味論は `out(T) = in(T-2) != in(T-1) ? 15 : 0` である。
入力が回路へ反映される 1 tick と、隣接 2 サンプルの比較による 1 tick の合計 2 tick の履歴に乗るので、入力変化の 1 tick 後に 1 tick 幅のパルスが出る。

オブザーバはインラインチェーン素子専用で、トーチと同じく `reg p = o;` のように reg には置けない。
入力が定常(変化なし)なら出力は 0 に整定するので、`#init`(§7.2)が止まる(発振しない)。

#### 4.6.1 エッジ亜種(`op`, `on`, `oe`)

接尾字でエッジ判定だけを差し替えた亜種が 3 つある。
1 tick パルス、強度 15、インラインチェーン専用、reg 不可という性質は本体 `o` と共通である。

| 表記 | 検出対象 | 判定式 |
|---|---|---|
| `op` | 立ち上がりのみ | `out(T) = (in(T-2) == 0 && in(T-1) > 0) ? 15 : 0` |
| `on` | 立ち下がりのみ | `out(T) = (in(T-2) > 0 && in(T-1) == 0) ? 15 : 0` |
| `oe` | 2値エッジ | `out(T) = ((in(T-2) > 0) != (in(T-1) > 0)) ? 15 : 0` |

接尾字は Verilog の posedge / negedge / edge に対応する。
`oe` は 0↔正 のトグルだけを拾い、強度変化(例 5 → 10)を無視する点で本体 `o` と異なる。
`op` はワンショット起動(電源 ON で 1 発)、`on` は電源 OFF の検出に向く。

```rv
x - op - y;                    // x の立ち上がりの次 tick に y へ 1 tick パルス
```

接尾字は既存素子文字 `c`/`d`/`o`/`r`/`t` 以外から選んである。
たとえば `od` は「オブザーバ + ダスト」という既存の素子列であり、立ち下がりの接尾字には使えない。
亜種の綴りも素子列になるため、名前衝突ルール(§2.2)により `on` などは reg / wire / ポート名に使えない。
エッジ亜種はゲームのオブザーバにない拡張である(§11.5)。

## 5. ワイヤーと接続

回路の点と点を素子列でつなぐ構文を扱う。
基本構文(§5.1)、再利用可能なテンプレート wire(§5.2)、サイド入力を持つ点(§5.3)、回路の入れ子(§5.4)の 4 つの層に分けられる。

### 5.1 チェーン文

2 つの reg やポートを素子列でつなぐには、チェーン文を直接書く。
書式は `from -素子列- to ;` である。
wire 変数の宣言は不要で、素子列をその場に直接並べる。

```rv
x-ddr2brdccbr4d3-y;            // x と y を素子列でつなぐ(チャンクは連結可)
x-d-r2-t-y;                    // '-' で区切って書いてもよい
x-y;                           // 長さ 0 のチェーン(直結)
```

チェーンの先頭と末尾は宣言済みの reg やポートで、中間チャンクは素子か wire 名でなければならない。
1 つの reg に複数のチェーンが合流した場合、強度の最大値をとる(ゲーム仕様準拠)。

端点に `.side` を付けると、コンパレータやリピーターの reg のサイド入力に結線できる(§5.3)。
中間チャンクおよび始点には `.side` を付けられない。

### 5.2 wire(再利用可能な素子列)

`wire` は端点を持たない素子列のテンプレートで、チェーンの中間チャンクとして展開できる。
同じ wire を複数箇所で使うと、各箇所で独立した素子列として展開される(物理的に別物)。

```rv
wire seg;
seg = d2ccd2;                  // ダスト 2 + コンパレータ + ダスト 2 を定義
x - seg - m;                   // x を seg 経由で m へ
y - seg - m;                   // 同じ seg を y → m にも再利用(各々別インスタンス)
```

wire の代入(`w = ...`)は接続ではなく素子列の定義である。
そのため端点(reg やポート名)、`.side` を書くことはできない。

書ける素子列の例:

- 単一素子(`w = r4;`)
- 複数素子(`w = d4-cc-d4;` も `w = d4ccd4;` も可)
- 他の wire を含めた合成(`wrap = base-r-base;`)

ただし循環参照はエラーとなる(`w1 = w2;` かつ `w2 = w1;` のような構成)。
未定義(宣言のみで未代入)の wire をチェーンで使うのもエラー。
wire の再代入は警告を出し、最後の代入が採用される。

wire 自体はチェーンの端点にできない(端点は reg / ポートのみ)。

### 5.3 名前付きの点

サイド入力を持つ素子(コンパレータ、ロック可能リピーター)は、reg に代入して名前付きの点にすることで 3 端子の結線を行う。
「後ろ(back)」「横(side)」「前(out)」の 3 つの結線点を、それぞれ別のチェーンで駆動する。

#### 5.3.1 コンパレータのサイド入力

```rv
reg cmp = cd;                  // 減算コンパレータ(cc = 比較)
x - d3 - cmp;                  // cmp が終端 → 後ろ(back)入力
s - cmp.side;                  // cmp.side → 横(side)入力
cmp - y;                       // cmp が始端 → 前(out)出力
```

| 端子 | 書き方 | 役割 |
|---|---|---|
| back | `cmp`(チェーン終端) | 後ろ入力 |
| side | `cmp.side`(チェーン終端のみ) | 横入力 |
| out | `cmp`(チェーン始端) | 前出力 |

出力は 1 tick 遅延で、モードにより次のように決まる。

- `cd`(減算):`out = max(0, back − side)`
- `cc`(比較):`out = back >= side ? back : 0`

back と side それぞれに複数ワイヤーが合流した場合は強度の最大値をとる(ゲームの左右サイド max と一致)。

#### 5.3.2 リピーターロック

リピータも reg に代入して名前付きの点にすると、コンパレータと同じ 3 端子を持つ。
横入力が `> 0` の間、出力はロック直前の値で凍結され、後ろ入力が変化しても追従しない。
横入力が 0 に戻ると通常のリピーターに復帰する(ロックは 1 tick 反応)。

```rv
reg m = r;                     // ロック可能リピーター(r / r1-r4。遅延規則は同じ)
x - m;                         // m が終端 → 後ろ(back)入力
lk - m.side;                   // m.side → 横(ロック)入力
m - y;                         // m が始端 → 前(out)出力
```

3 端子の書き方と合流規則はコンパレータと同じである。

チェーン内インラインのリピーター(例 `x-r-y`)は従来どおりロックを持てない(横入力を引き出せない)。

#### 5.3.3 サイド入力に共通の制約

コンパレータ reg とリピーター reg のサイド入力には次の共通制約がある。

- `.side` を始端(信号源)に使うのはエラー(横は入力専用)
- `.side` をコンパレータでもリピーターでもない reg に付けるのはエラー
- 修飾子は plain のみで、信号強度リテラルや `const` / `mutable` は付けられない(§3.4)
- 宣言時初期化に限る(`reg cmp; cmp = cd;` のような後置代入は不可、§3.4)

バス宣言 `reg[W] m = r;` では、この 3 端子がレーンごとに展開される(§6.5)。

### 5.4 階層インスタンス化

`logic` の本体から別の `logic` を呼び出すことで、回路を入れ子に組める。

```rv
logic AND2(input x1, input x2, output y) {
    reg na, nb, s;
    na = NOT(x1);         // out = callee(args...)
    nb = NOT(x2);
    s  = OR2(na, nb);
    y  = NOT(s);          // x1 & x2 = !( !x1 | !x2 )
}
```

書式は `out = callee(args...);` である。
`out` はこの logic 内の reg か出力ポートを、`args` は reg か入力ポート名を指す(wire や素子、数値リテラルは渡せない)。
`out` と `args` には、バスのレーン `x[k]` やスライス `x[hi:lo]`(添字は §6.3.1 の定数式)、連結 `{a, b, c[1:0]}`(§6.3)も書ける。

結線の意味は次のとおり。

- `out` には下位 logic の出力ポートが減衰なしで直結される
- `args` の各ノードは下位 logic の入力ポートを駆動する
- 下位インスタンスは sim 変数とは無関係に回路として展開される

出力ポートが 1 個ならこの形(`out = callee(...)`)で受ける。
複数の出力ポートを別々の端点で受け取りたいときは §5.5 のタプル束縛を使う。
1 個の出力ポートはスカラでもバス(`output[N]`)でもよく、バスの場合はレーン対応で同種の多出力として束縛される(§6)。

引数の個数は下位 logic の入力ポート数と一致しなければならない。
各引数は reg やポート、バス全体、バスのレーン `x[k]` / スライス `x[hi:lo]`、または連結 `{e1, e2, ...}` である。
レーンはスカラ入力ポートへ、スライスは同幅のバス入力ポートへ束縛される。
連結は幅 = 各要素の和のバス引数になり、同幅のバス入力ポートへ束縛される(スカラ数本を束ねてバスポートへ渡せる)。
連結要素の制約はチェーン端点(§6.3)と同じで、加えてネスト呼び出し(§5.6)も要素には書けない。

スライス・連結のレーン列は §6.3 の並び順のまま、ポートの昇順レーン `[0..W)` と element-wise に対応する。
チェーンでバス全体名につないだ場合と同じ規則で、identity で渡すなら昇順 `x[0:3]`(バス全体 `x` の部分版)、降順 `x[3:0]` はビット反転になる。
連結では先頭要素がポートのレーン 0 に対応する。

`out` 側の連結は逆向きの分配で、バス出力ポートのレーン列を並び順に各要素へ受ける。

```rv
{rest, msb} = SHIFT4(x);       // 幅 4 の出力を rest(下位 3 レーン)と msb に分配
```

再帰インスタンス化(自己呼び出し、相互の循環呼び出し)はエラーとなる。

### 5.5 多出力 logic とタプル束縛

`logic` は **複数の出力ポート** を持てる。
半加算器の `sum` と `carry` のように、同じ入力から別々の意味を持つ出力を同時に取り出したいときは、出力ポートをポート列にそのまま並べる。

```rv
logic HALF_ADDER(input x1, input x2, output sum, output carry) {
    sum   = XOR2(x1, x2);
    carry = AND2(x1, x2);
}
```

多出力 logic を呼び出すときは、束縛先を `(t1, t2, ...)` のタプルで受ける。
出力ポートと位置対応で、レーン対応で減衰なしに結線する。

```rv
module m {
    var x1, x2, sum, carry;
    sim {
        x1 = 0; x2 = 0;
        (sum, carry) = HALF_ADDER(x1, x2);    // 多出力タプル束縛
        #init
    }
}
```

タプル束縛は logic 本体内でも sim ブロック内でも書ける。
書式と制約は次のとおり。

- 各 target は reg / ポート / バス reg / バスポート / var / バス var、バスのレーン `sum[k]` / スライス `sum[hi:lo]`(添字は §6.3.1 の定数式)、または連結 `{c, s[3:0]}`(§5.4 の `out` 側と同じ分配)。`.side` は付けられない
- target 数と callee の出力ポート数は厳格に一致しなければならない(過不足ともエラー)
- 同じレーンを 2 度以上束縛するのはエラー。字面の重複(`(p, p)`)だけでなく、解決後レーンの部分重複(`(sum[1:0], sum[1])` や `(sum, sum[0])`)、連結内部の重複(`{p, p}`)も検出する
- 各 target と対応出力ポートは、形(スカラ / バス)と幅が一致しなければならない
- 1 個の出力ポートに対しても `(t) = callee(args...)` の **1-target タプル形** を許容する(従来の `t = callee(args...)` と等価)

引数列・再帰禁止・束縛のセマンティクスは §5.4 と同じである。
レーンを引数と target に直接書けるので、全加算器のリップルキャリー接続のような多ビットの繰り返し構造が 1 段 1 行で書ける。

```rv
logic ADD4(input[4] a, input[4] b, input c_in, output[4] sum, output carry) {
    reg c0, c1, c2;
    (sum[0], c0)    = FULL_ADDER(a[0], b[0], c_in);   // レーンを引数・target に直接書く
    (sum[1], c1)    = FULL_ADDER(a[1], b[1], c0);
    (sum[2], c2)    = FULL_ADDER(a[2], b[2], c1);
    (sum[3], carry) = FULL_ADDER(a[3], b[3], c2);
}
```

### 5.6 引数のネスト呼び出し

logic 呼び出しの引数には、名前の代わりに **別の logic 呼び出し** を直接書ける。

```rv
logic MUX2(input in0, input in1, input sel, output y) {
    y = s_or(s_and(in0, s_not(sel)), s_and(in1, sel));
}
```

ネストした呼び出しの出力ポートは、外側の入力ポートへ減衰なしで直結される。
これは中間 reg / var を宣言して受け渡した場合と等価な配線であり、上の例は次の展開形と同じ回路になる。

```rv
reg ns, t0, t1;
ns = s_not(sel);
t0 = s_and(in0, ns);
t1 = s_and(in1, sel);
y  = s_or(t0, t1);
```

規則は次のとおり。

- logic 本体(§5.4)でも sim(§7.3)でも書ける
- ネストできるのは **出力ポートちょうど 1 個** の logic に限る(多出力はタプル束縛で一旦受けてから渡す)
- ネストの深さに制限はない。再帰禁止(§5.4)はネスト経由でも同様に検査される
- 引数には `#(...)` のジェネリック実引数(§8.4)も付けられる(`g(h#(W=8)(x))`)
- 出力ポートと入力ポートの形(スカラ / バス)と幅は一致しなければならない
- `scan()`(§7.8)は logic ではないためネストできない

中間点は名前を持たないため、トレース(`-t`)や `monitor` では観測できない。
途中の値を観測したい回路では、従来どおり中間 reg / var を宣言して受ける。

## 6. バス

複数のレーンを束ねた配線を一行で書ける糖衣構文を扱う。
バスはエラボレーション時に N 本のスカラチェーンへ展開されるだけで、シミュレーションエンジン自体はバスを知らない。
このため合流の max、順序素子、`#init` の挙動はすべてスカラ点と完全に同一である。

### 6.1 バスの宣言と添字

`reg[N] name;` は N 本の並列レーンを宣言する。
各レーン `name[0]` から `name[N-1]` は通常のスカラ点として使える。

```rv
reg[4] in;                     // バス宣言:in[0]..in[3]
reg[4] buf;
x - in[0];                     // レーンを添字 [k] で個別に指定
in - r - buf;                  // バス全体:1 行が 4 レーンぶん(各レーンに独立したリピータ)に展開
buf[0] - y;                    // レーンを取り出して出力ポートへ
```

| 書き方 | 意味 |
|---|---|
| `reg[N] a;` | 幅 N のバス宣言。`[N]` は `reg` と名前の間に置く。`N` はリテラルか定数式(§8.3) |
| `a[k]`(k は定数式、§6.3.1) | レーン k。スカラ点と同じに使える(0 ≦ k < N) |
| `a`(チェーン端点、添字なし) | バス全体。同幅の相手と element-wise に接続 |

バス名(ベース名)も §2.2 の名前の制約を受ける。
バスはチェーンの端点専用で、中間チャンクには置けない。

複数の宣言を 1 行に書く(`reg[N] a, b;`)場合、`[N]` は宣言列全体に適用される。

### 6.2 バスチェーンとバスポート

バスチェーンは両端のバスを element-wise に接続する糖衣である。
両端の幅が一致していれば、中間の素子列が各レーンごとに独立して展開される。
中間 wire 名もレーンごとに別インスタンスへ展開される。

ポートもバスにできる。
`input[N] x` / `output[N] y` は N レーンの並列ポートで、本体では内部バス reg と同じく添字 `x[k]` やバス全体 `x` で使える。

```rv
logic AND2(input[4] x1, input[4] x2, output[4] y) {
    reg[4] nx1;  reg[4] nx2;  reg[4] s;
    x1 - t - nx1;              // nx1 = NOT x1(4 レーン一括)
    x2 - t - nx2;
    nx1 - s;  nx2 - s;         // s = nx1 OR nx2(各レーンで max 合流)
    s - t - y;                 // y = NOT s = x1 AND x2(ビット単位 AND)
}
```

バス出力ポートは同種の多出力(N レーン)である。
`out = g(args...)` の `out` がバス var ならレーン対応で束縛される(§7.3)。
引数と出力先のポートは、形(スカラとバスの区別)と幅が一致しなければエラーとなる。

### 6.3 スライス `x[hi:lo]` と連結 `{x, y}`

チェーンの端点には、レーンの部分範囲(スライス)や複数端点の連結を書ける。
どちらもバス糖衣の範囲で完結し、レーン列へ展開するだけである。
両端の幅(レーン数)が一致すれば element-wise に接続される。

```rv
x[3:0] - r - y[3:0];           // 部分バスを素子列でつなぐ
x[3:0] - y[0:3];               // 向きを変えるとビット反転配線
hi - {carry, sum};             // 連結を一括配線(幅は各要素の和)
{x[2:0], x[3]} - out[3:0];     // 連結 + スライスで 1 ビット左ローテート
```

| 書き方 | 意味 |
|---|---|
| `x[hi:lo]` | スライス(包含)。`hi >= lo` で降順 `[hi, hi-1, .., lo]`、`hi < lo` で昇順 `[hi, .., lo]`。`x[k:k]` は `x[k]` と等価(幅 1)。`hi` / `lo` は範囲内の定数式(§6.3.1) |
| `{e1, e2, ...}` | 連結。各要素 `ei` はスカラ点、レーン `x[k]`、スライス `x[hi:lo]`、バス全体のいずれか。左から順にレーンを連接し、幅は各要素の和 |

バス全体名はインデックス昇順 `[0..N)` でレーンを供給する(既存のバス全体チェーンと同じ)。
ビット順を制御したいときはスライス `x[hi:lo]` で明示する。

スライスと連結は、チェーンの端点のほか logic 呼び出しの引数と束縛 target(§5.4・§5.5・§7.3)にも書ける。
チェーンの中間チャンクや素子代入の `=` には置けない。
連結要素に `.side` やネストした `{{..}}` は使えない(呼び出しの引数 / target ではネスト呼び出しも不可)。
範囲外添字や非バスのスライスはエラーとなる。

### 6.3.1 添字の定数式

レーン添字 `a[k]` とスライス添字 `x[hi:lo]` の添字には、リテラルに加えて **定数式** を書ける。
使える要素はバス幅 `[expr]`(§8.3)と同じで、リテラル、`param`、数値 `#define`、`+ - * / %`、単項 `-`、括弧である。

```rv
param N = 2;

logic SPLIT #(W=8)(input[W] x, output[W/2] hi_o, output[W/2] lo_o) {
    x[W-1:W/2] - hi_o[W/2-1:0];    // 上位半分
    x[W/2-1:0] - lo_o[W/2-1:0];    // 下位半分
}

logic TAP(input[4] a, output y) {
    a[N+1] - y;                    // param 参照はパース時に a[3] へ確定
}
```

ジェネリック param(§8.4)を含む式は幅式と同じくインスタンス化時に評価され、含まない式はパース時に確定する。
評価結果には降順 / 昇順・幅 1 の規則(§6.3)がそのまま適用される。
評価結果が範囲外ならエラーとなる(ジェネリック param を含む式では、インスタンス化されて初めて検出される)。

### 6.4 バスとスカラの直結(ブロードキャスト)

バスチェーンの片側がスカラ(幅 1)のとき、もう片方の幅 N へブロードキャストして接続する。
これもレーン列へ展開するだけのバス糖衣である。

```rv
reg[4] x;  reg y;
x - y;                         // fan-in:x の 4 レーンすべてを y へ直結
y - x;                         // fan-out:y を x の 4 レーンすべてへ直結
x - r - y;                     // 素子列を挟む形も可
```

- **fan-in**(`bus[N] - scalar`):N レーンが 1 点へ合流する。接続は MAX 合流なので `scalar = max(全レーン)`
- **fan-out**(`scalar - bus[N]`):1 点が N レーンすべてを駆動する(各レーンが同じ源)

接続は有向で、`-` の左が源、右が行き先である。
これは「複数の信号を 1 本にまとめる」「1 つの源を複数点へ複製する」操作を、有向モデル上の糖衣として書けるようにしている(ゲームのダストは双方向だが、本処理系の接続は逆流しない、§11.2)。

スカラ側はスカラ点、単一レーン `a[k]`、幅 1 のスライスや連結のいずれでもよい。
相手側はバス全体、スライス、連結のいずれでもよい(どちらかが幅 1 ならブロードキャスト)。
両端とも幅 >1 で不一致のときはエラーとなる(`bus[4] - bus[2]` など)。
1 レーンだけ繋ぎたいときは添字 `name[k]` でスカラにする。

### 6.5 素子代入つきバス reg(コンパレータ / リピーター)

`reg[W] m = r;` は、ロック可能リピーターの「名前付きの点」(§5.3)を W レーンぶん一括で宣言する。
コンパレータ(`cc` / `cd`)も同様で、各レーンが独立した back / side / out の 3 端子を持つ。

```rv
reg[4] m = r;                  // ロック可能リピーター 4 レーン
x  - m;                        // back:同幅バスとレーン対応(スカラなら broadcast)
lk - m.side;                   // side:1 本のロック線を全レーンへ broadcast
m  - y;                        // out:各レーンの出力
```

チェーン端点の解釈はスカラの「名前付きの点」と同じで、終端の無印はレーンごとの back、始端は out、`.side` は横入力である。
`.side` は 3 つの粒度で書ける。

| 書き方 | 意味 |
|---|---|
| `m.side` | 全レーンの side。源がスカラなら broadcast、同幅バスなら element-wise(§6.4 と同じ規則) |
| `m[k].side` | レーン k の side だけを駆動 |
| `m[hi:lo].side` | スライス範囲のレーンの side(§6.3 の降順 / 昇順規則が効く) |

これで、ロック線を共有する W 本の並列リピーター(= W ビットレジスタの保持部)や、バス同士のレーン単位減算(`reg[W] c = cd;` + 同幅 side)が 1 宣言で書ける。
side を繋がないレーンはスカラと同じく side = 0 として動く(リピーターは非ロック、コンパレータはパススルー)。

制約もスカラの「名前付きの点」と同じである(§3.4、§5.3.3)。
plain のみ・宣言時初期化に限り、`r0` と信号強度リテラル、コンパレータ / リピーター以外の素子は書けない。

### 6.6 未対応の項目

次の項目は将来拡張として残してある(現時点ではエラーや非対応となる)。

- バスの `const` / `mutable` 宣言と、素子代入(§6.5)以外の初期化子
- バスを素子代入の左右(`a = 15r` など)に置くこと
- wire 素子列のバス化、`wire[N]` 構文
- 異種多出力のタプル束縛

## 7. sim ブロック

`module` 内の `sim { ... }` は、回路を駆動して観測するテストベンチを書く逐次インタープリタ言語である。
変数代入、時間進行、観測の 3 種類の文を中心に、`if` / `while` / `for` の制御構文も書ける。

### 7.1 全体構造

```rv
module test {
    var x, y;
    sim {
        x = 0;
        y = NOT(x);       // インスタンス化と入出力束縛
        #init                  // 定常状態まで待つ
        x = 10;
        #1                     // 1 tick 進める
        ?monitor("時刻=% | x=% | y=%\n", $time, x, y);
    }
}
```

`sim` の最初の処理は通常「変数の初期化と束縛」、続いて「時間を進めて観測」となる。
`?monitor` は sim 開始時に登録され、全ウェイト終了時(以下の`#`付き文の終了時)で自動発火する(§7.4)。

### 7.2 時間の進め方

| 文 | 振る舞い |
|---|---|
| `#n` | n tick 進める。`$time` を進めて `?monitor` を発火 |
| `#init` | 更新キューが空(定常状態)になるまで待つ。`INIT_TIMEOUT`(§8.1)超過でエラー |
| `#until(cond)` | `cond` が真(`!= 0`)になるまで tick を進めるイベント駆動待機 |
| `wait(n)` | `$time` を進めず、`?monitor` も発火させずに n tick 待つ(発振回路向け) |

`#until` は出力が立つまでの tick 数を知らずに「立つまで」待てる便利な待機文である。
`$time` は `#n` と同様に進み、完了時の `$time` が経過 tick となる。
条件が最初から真なら 0 tick で抜ける(`while` と同じく進入時に評価する)。
`INIT_TIMEOUT` 超過でエラーとなり、発振や永遠に成立しない条件を検出できる。

`$time` は `#init` 完了時(`#init` を使わない sim では開始時)を 0 とした経過 tick である。

### 7.3 入出力の束縛

`v = logic名(変数...)` で回路をインスタンス化し、変数を入出力に束縛する。

```rv
y = NOT(x);               // x が入力、y が出力に束縛される
```

束縛の規則は次のとおり。

- 同じ「logic 名と引数列」の組合せは同一インスタンスを共有する(target はキャッシュキーに含めない)
- 引数の添字・連結はキーに含まれる。`g(x[0])` と `g(x[1])` は別インスタンスで、`g(x[0])` の 2 度目の呼び出しは同一インスタンスを共有する(`x[k:k]` は `x[k]`、単一要素の連結 `{x}` は `x` と同じキーに正規化される)
- 入力変数は継続的に束縛され、出力変数は毎 tick 更新される
- バスポート(§6.2)には、形と幅が一致するバス var を丸ごと渡す

バス var を含む例:

```rv
var[4] x, y;
y = AND2(x, x);                // バス var を丸ごとバスポートへ束縛
y[0] = NOT(x[3]);              // レーンを引数・target に直接束縛
```

スカラポートにはスカラ var、バスポートにはバス var を渡す必要があり、形や幅が一致しないとエラーとなる。
引数と target にはバス var のレーン `x[k]` / スライス `x[hi:lo]`、連結 `{a, b, c[1:0]}` も書ける(§5.4 / §5.5 と同じ規則)。
連結を使うと、スカラ var 数本を束ねてバスポートへ渡したり(`y = g({a, b, c});`)、バス出力を複数の var へ分配したり(`{rest, msb} = g(x);`)できる。
束縛は sim の実行前に静的に確定するため、添字は §6.3.1 の定数式に限る(通常のレーン代入 `x[i] = v` と違い、実行時の var 添字は書けない)。

多出力 logic を sim から呼ぶときは、束縛先を `(t1, t2, ...)` のタプルで受ける(§5.5)。

```rv
var x1, x2, sum, carry;
(sum, carry) = HALF_ADDER(x1, x2);   // 2 出力をタプルで受ける
```

target 数と出力ポート数は厳格に一致(過不足ともエラー)。
同一 target を 2 度書くこと(`(p, p) = ...`)もエラー。
1 出力 logic でも `(v) = callee(...)` の 1-target タプル形を許容する(従来の `v = callee(...)` と等価)。

引数には別の logic 呼び出しを直接ネストできる(§5.6)。

```rv
y = s_or(s_and(x1, x2), s_xor(x3, x4));   // y = (x1 & x2) | (x3 ^ x4)
```

インスタンス共有の規則はネストした部分式にも及ぶ。
上の `s_and(x1, x2)` は、同じ引数列の standalone 呼び出し(`t = s_and(x1, x2);`)と同一インスタンスを共有する。

回路へ束縛した変数は、入力反映時に 0 から 15 にクランプされる。
範囲外の値を代入した場合、変数ごとに 1 回だけ警告が出る(§10)。

### 7.4 観測

観測は 3 種類ある。
ゴールデンテスト向けの `monitor`、自己検証向けの `assert` と `expect` である。

#### 7.4.1 `?monitor` と `monitor`

`?monitor(fmt, ...)` は sim 開始時に登録される自動発火型の観測文である。
すべてのウェイト(`#init` / `#n` / `#until`)が完了した直後に自動実行される。
sim 内のどこに書いても全時刻が出力される(Verilog の `$monitor` 風)。

`monitor(fmt, ...)` は文の位置で 1 回だけ出力する命令文である。
`?` の有無で「自動発火」「その場で発火」を区別する。

書式文字列の規則は次のとおり。

- `%` は次の引数を 10 進で消費する
- `%N`(例 `%2`)は最小幅 N(右寄せスペース詰め)を指定する
- `%0N`(例 `%04`)は最小幅 N(右寄せ **ゼロ埋め**)を指定する
- `%%` は `%` リテラルを表す
- 隣接プレースホルダは `%%` がエスケープと衝突するため、幅付き(例 `%1%1%1%1`)で書く

型サフィックスで基数を切り替えられる。

| 書式 | 意味 | `monitor("%s\n", 10)` の例 |
|---|---|---|
| `%b` | 2 進 | `1010` |
| `%x` | 16 進(小文字) | `a` |
| `%o` | 8 進 | `12` |

幅とゼロ埋めは基数指定にも前置できる(`%Nb` / `%0Nb` / `%Nx` / `%0No` など)。
`%4x` は幅 4 で右寄せスペース埋め、`%04x` は幅 4 で右寄せゼロ埋めとなる(`monitor("'%04x'", 255)` → `'00ff'`)。

負の値は `-` 接頭 + 絶対値を基数表記する。
幅指定は符号を含めて数え、ゼロ埋めは符号を先頭に保ったままゼロを挿入する。
たとえば `monitor("'%04b'", -3)` は `'-011'` を出す(redv は HDL でも値の幅情報を持たないため、64 bit ラップではなく読みやすさを優先した)。

型サフィックス `%d` / `%t` は廃止済み(指定するとエラーになる)。
10 進は `%` 単独で書く。

##### バス var を 1 引数で渡す

引数にバス var(添字なし、§7.6)をそのまま渡すと、各レーン強度 0–15 を 4 bit のニブルとして下位から順にパッキングし、1 個の整数として表示する。
`lane[0]` が最下位ニブル、`lane[N-1]` が最上位ニブルになる。
レーン添字を 1 つずつ書かずに、バス全体を 1 行で表示できる。

```rv
var[4] bus;                    // バス幅 4
bus[0] = 15;  bus[1] = 0;  bus[2] = 15;  bus[3] = 8;
// packed = 8*16^3 + 15*16^2 + 0*16^1 + 15*16^0 = 0x8F0F = 36623
monitor("%x % %b\n", bus, bus, bus);
// → "8f0f 36623 1000111100001111"
```

`%x` / `%b` ではレーン境界を保つため、既定で N 桁 / 4N bit にゼロ埋めする。
ユーザー指定幅はそれを上回るときだけ働く下限として扱う。

| 書式 | bus[4]={[3]=8,[2]=15,[1]=0,[0]=15} の出力 |
|---|---|
| `%x` | `8f0f`(既定 N=4 桁) |
| `%8x` | `    8f0f`(下限 8、スペース埋め) |
| `%08x` | `00008f0f`(下限 8、ゼロ埋め) |
| `%b` | `1000111100001111`(既定 4N=16 bit) |
| `%` | `36623`(10 進合成) |
| `%o` | `107417`(8 進合成、レーン境界とは独立) |

バスは無符号として表示する。
全レーンに 15 が立った bus[16] のような場合でも符号反転せず `ffffffffffffffff` などを返す。

バス var を **書式引数の最上位**として渡したときだけこの合成を行う(`monitor("%\n", bus)`)。
`bus + 1` のように式の中に置くと従来どおり「インデックスが必要」とエラーになる。
これは「バス全体を 1 個の値として扱う」用途を monitor の引数位置に限定し、ほかの式での誤用を防ぐためである。
レーン単独を扱いたいときは `bus[k]` を使う。

バス幅 N > 16 は i64 に収まらないためエラーとなる。
そのときはレーンを 1 つずつ表示する。

#### 7.4.2 `assert` と `expect`

`assert(cond)` は `cond` が偽(`= 0`)なら失敗を stderr に記録する。
失敗時は行番号と式を表示する。

`expect(actual, expected)` は両者が不一致なら「実際の値と期待値」を stderr に記録する。

いずれも失敗しても sim は止まらず継続する。
全 module の sim 実行後に集計し、合格なら `N assertion(s), all passed`、失敗があれば `assertions: M of N failed` を出して非ゼロ終了する。

これにより、`monitor`(stdout)に頼らず終了コードで合否が分かる自己検証テストベンチを書ける(`examples/assert_selfcheck.rv`)。

### 7.5 制御構文と式

`sim` ブロック内では C 風の制御構文を使える。

```rv
for (i = 0; i < 4; i = i + 1) {
    x = i;
    #1
}
```

`if` / `while` / `for` の文法は C 準拠で、条件式は通常の整数式である。
比較や論理演算子も使える。

`var` は整数変数(C の `int` 相当)で、回路には組み込めない。
sim 式の演算は通常の整数演算であり、回路のレッドストーン強度(0 から 15)とは独立に動く。
ただし回路へ束縛した変数は入力反映時に 0 から 15 にクランプされる(§7.3)。

### 7.6 バス var

`var[N] x;` で N 本のレーンを持つバス var を宣言する。

```rv
var[4] x;
x[0] = 15;                     // 個別のレーンに代入(添字は実行時に評価する整数式)
x = 0;                         // 全レーンへブロードキャスト(一括クリア)
```

バス var の使い方の規則は次のとおり。

- レーン読み書きは `x[k]` の形で行う(添字は実行時評価の整数式、`for` ループ変数も使える)
- 添字なしの `x` をスカラ式に使うとエラー(レーン指定が必要)
- `x = 式;` は全レーンへブロードキャストする(`x = 0;` で一括クリアできる)

### 7.7 パルス代入とクロック生成

#### 7.7.1 パルス代入

`v = 式 ~ w;` はパルス代入である。
`v` に値を入れた後、`w` tick 経過すると自動的に 0 に戻す(代入 → ウェイト → 0 代入 を手で書かずに済む)。

```rv
x = 10 ~ 3;                    // x = 10 を 3 tick 維持してから x = 0
```

`w` は 1 以上の整数(式可、評価は代入時)である。
`w` tick の数え方は、実行されたあらゆる tick(`#n` / `wait(n)` / `#init`)を含む。

期限到来前に同じ var を通常代入すると、保留中パルスは解除される。
パルス代入で上書きすれば新しい幅で置き換わる。

#### 7.7.2 クロック生成

`clock(var, N)` はテスト用クロックを 1 行で生成するシュガーである。
スカラ var を「各レベル N tick 保持」で 0 と 15 に自動トグルする(full period = 2N、50% デューティ)。

- 呼び出し直後は Low(0)
- 以降 N tick ごとに反転
- `clock()` 自体は時刻を進めず、後続の `#n` / `wait` / `#init` / `#until` が tick を刻む間にトグルする

同じ var への通常代入でクロックは解除される。
`N` は 1 以上の整数(式可)である。
`var` は宣言済みのスカラ var で、バス var は使えない。

monitor で読む var の値は「次 tick に適用される値」を映すため、1 tick 先行して見える(パルス代入と同じ挙動)。
デューティ比や初期位相の指定は将来拡張である(`examples/clock_sugar.rv`)。

### 7.8 入力読み込み(`scan`)

`v = scan()` は stdin から空白や改行区切りの整数を 1 つ読み、変数 `v` に代入する。
EOF(入力切れ)や非数値はエラーとなる。
回路へ束縛した変数は読み取り後も 0 から 15 にクランプされる。

書式を渡すと入力側の基数を指定できる(monitor の出力書式と対称)。
書式は `%[bxo]?` ちょうど一つで、リテラル文字や幅指定は受け付けない。

| 書き方 | 入力の基数 |
|---|---|
| `scan()` / `scan("%")` | 10 進 |
| `scan("%b")` | 2 進 |
| `scan("%x")` | 16 進(`a`–`f` / `A`–`F` 両方 OK) |
| `scan("%o")` | 8 進 |

いずれの基数でも、先頭の `-` または `+` は符号として読む。
基数に属さない文字に当たった時点でトークンを区切る(`scan("%x")` で `ff\n` を読むと `255` を返し、改行は次の `scan` まで残す)。

テストでは `redv foo.rv < input.txt` のように入力を固定すれば、決定的に再現できる。

## 8. ディレクティブ

### 8.1 `#define`

```rv
#define INIT_TIMEOUT 200       // #init のタイムアウト(既定 1000 tick)
#define BURNOUT_LIMIT 8        // トーチ焼き切れ:窓内トグル回数(既定 8)
#define BURNOUT_WINDOW 30      // 同:監視窓(既定 30 tick)
#define BURNOUT_COOLDOWN 30    // 同:強制 OFF 期間(既定 30 tick)
#define MODE element           // 素子レベルモード(logic モードは将来対応)
```

数値の `#define` はバス幅や sim 式から名前で参照できる(`param` と同じ表に入る、§8.3)。

値は `param` と同じ定数式を受理する。
リテラル、既出の `#define` / `param`、`+ - * / %`、単項 `-` / `!`、括弧を組み合わせられる。

```rv
#define W 4
#define N (W * 2)              // 算術:8
#define HALF (N / 2)           // ネスト:4
#define ONES (0b1111)          // 2 進 / 16 進リテラルとの混在(§1.3)
```

`MODE` のみは将来のモード切替え用の予約名で、識別子値だけを受理する(`element` 以外は警告)。
未定義の名前を参照したり、ゼロ除算するとエラーとなる。

### 8.2 `#include`

```rv
#include "other.rv"            // ファイル取り込み
```

#### 8.2.1 バンドル済み標準ライブラリ

`#include "stdlogic"` のように **バンドル名** を渡すと、redv バイナリに埋め込まれた標準ライブラリを取り込む。
ファイルを別途用意したり相対パスを書いたりする必要はなく、どこから `redv` を実行しても同じソースが読まれる。

| 名前 | 内容 |
|---|---|
| `stdlogic` | 基本論理ゲート(`s_not` / `s_and` / `s_or` / `s_xor` / `s_nand` / `s_nor` / `s_xnor`)。全ゲートがジェネリック幅 `#(W=1)` |
| `stdmem` | ラッチ・レジスタ(`s_rslatch` / `s_dlatch` / `s_dff` / `s_register`)。`s_rslatch` 以外はジェネリック幅 `#(W=1)`。内部で `stdlogic` をネストして取り込む |

同じバンドル名を 1 ソース中で複数回 `#include` しても、2 度目以降は no-op になる(重複定義エラーにはならない)。
ネストした include 経由でも同じ判定が効く。
ファイル include の方は従来どおり重複判定をしないので、循環や 2 度取り込みは利用側で避ける。

バンドル名と一致しない名前は通常のファイル include へフォールバックする。
たとえば `#include "stdfoo"` はバンドル一覧に無いので、`stdfoo` / `stdfoo.rv` を相対パス・絶対パスで探し、見つからなければエラーとなる。

```rv
#include "stdlogic"

logic EQUAL(input x1, input x2, output y) {
    y = s_xnor(x1, x2);        // 1 ビット一致検出 = XNOR
}
```

`s_xor` / `s_xnor` は内部で `s_not` / `s_and` / `s_or` を呼ぶ階層構造で、合計 4〜5 tick の遅延を持つ。
ゲートごとの伝搬遅延は `examples/stdlogic_demo.rv` のヘッダコメントを参照。

全ゲートはジェネリック幅 `#(W=…)`(§8.4)を持ち、全ポートが W レーンのバスポートになる。
演算はレーンごとの element-wise(ビット単位)である。
既定の `W=1` はスカラ互換で、上の例のようにスカラ var / reg をそのまま結線できる。

```rv
var[4] a, b, y;
y = s_xor#(W=4)(a, b);         // 4 ビットのビット単位 XOR
```

`stdmem` はラッチ・レジスタの 4 素子を提供する。

| logic | ポート | 動作 |
|---|---|---|
| `s_rslatch` | `(set, reset) -> (q, nq)` | RS ラッチ(NOR 2 個のたすき掛け)。`set` で q=15、`reset` で q=0、両方 0 で保持 |
| `s_dlatch` | `#(W=1)` `(x, en) -> q` | D ラッチ。`en > 0` で透過、`en = 0` で保持(ロック付きリピーター) |
| `s_dff` | `#(W=1)` `(x, clk) -> q` | D フリップフロップ。`clk` の立ち上がりで `x` を取り込む |
| `s_register` | `#(W=1)` `(x, ld, clk) -> q` | ロードイネーブル付きレジスタ。立ち上がりでも `ld = 0` なら保持 |

`s_dlatch` / `s_dff` / `s_register` はジェネリック幅 `#(W=…)`(§8.4)を持つ。
データ経路(`x` / `q`)だけが W レーンのバスポートで、制御線(`en` / `ld` / `clk`)はスカラのまま全レーンに共有される(内部は §6.5 のバス named point + broadcast)。
既定の `W=1` はスカラ互換で、従来どおりスカラ var / reg をそのまま結線できる。

```rv
var[4] x, q;  var ld, clk;
q = s_register#(W=4)(x, ld, clk);   // 4 ビットのロードイネーブル付きレジスタ
```

`s_dff` / `s_register` は、クロックの立ち上がり検出(オブザーバ `op`、§4.6.1)で
ロック付きリピーターを 1 tick だけ解錠する構成である。
取り込みはエッジの 3 tick 後に行われるため、`x`(`s_register` では `ld` も)は
エッジの直前 1 tick から 4 tick 後まで安定させる必要がある。
また `s_rslatch` は `set = reset = 0` のままだと初期状態が定まらず発振するので、
最初の `#init` の前に `set` か `reset` を一度 15 にして初期状態を確定させる。
各素子の伝搬遅延は `src/stdlib/stdmem.rv` のヘッダコメントを参照。

`stdmem` は取り込み時に `stdlogic` をネストして取り込む。
利用側が別途 `#include "stdlogic"` と書いていても、前述の重複判定により 2 度目は no-op になる。

### 8.3 `param`(パラメータ定数)

`param NAME = <定数式>;` でトップレベルに整数定数を宣言する。
バス幅や sim 式から名前で参照でき、幅をリテラル固定せず 1 定義を複数幅で再利用できる。

```rv
param W  = 4;                  // 定数
param W2 = W + 1;              // 既出の param を使った定数式も可(前方参照は不可)

logic NOT(input[W] x, output[W] y) { x - t - y; }   // 幅を param で

module m {
    var[W] x;                  // バス幅に param
    var[W2] y;                 // 定数式の結果(= 5)幅
    var i;
    sim {
        x = 0;
        for (i = 0; i < W; i = i + 1) { x[i] = 15; }   // sim 式でも参照可
    }
}
```

`param` の解決規則は次のとおり。

- 値の解決はパース時に行う。`param` と数値 `#define` を同じ表に持ち、`[ ... ]` 幅と sim 式の両方がこの表を引く
- 幅 `[expr]` は定数式を受理する(リテラル、param、`+ - * / %`、単項 `-`、括弧)
- 評価結果は 1 以上でなければならない(0 以下はエラー)
- 定数式に使えないもの:`$time`、バス添字、比較や論理演算子、未定義名
- sim 式では同名の `var` が優先される(param を隠す)。`var` にない名前のときだけ param を引く

### 8.4 logic ごとのジェネリック幅

`logic name #(P=既定値, ...)(...)` 形式で、logic ごとに **ジェネリック幅 param** を宣言できる。
呼び出しごとに値を変えて、同じ logic 定義を異なる幅のインスタンスとして展開できる。

```rv
logic NOT #(W=4)(input[W] x, output[W] y)
{
    reg[W] s;
    x - s;
    s - t - y;
}

module m
{
    var[4] x4, y4;
    var[8] x8, y8;
    sim {
        x4 = 0;  x8 = 0;
        y4 = NOT(x4);            // 既定値 W=4 で展開
        y8 = NOT#(W=8)(x8);      // 実引数 W=8 で別インスタンス
        #init
    }
}
```

宣言と呼び出しの規則は次のとおり。

- 宣言: logic 名の直後・ポート列の前に `#(P=expr, Q, ...)` を書く。各 param には省略可能な既定値を `=` で与えられる
- 既定値の式は `param` と同じ定数式(`eval_const`)で評価し、パース時に確定する(他のジェネリック param は参照できない)
- 呼び出し: `callee#(P=expr, ...)(args)` の形で実引数を渡す。`#(...)` を書かなければ全 param で既定値が使われる
- 実引数の式は呼び出し側のスコープで評価する。トップレベル `param` / 数値 `#define` のほか、呼び出し側 logic 自身のジェネリック param も参照できる(階層パススルー)
- 既定値も実引数も無い param で呼び出すとエラー

logic 本体内では、宣言された param 名を `input[W]` / `reg[W+1]` などの幅式と、`x[W-1]` / `x[W-1:W/2]` などのレーン / スライス添字(§6.3.1)に使える。
param を参照する式は呼び出し時まで遅延し、各インスタンスで個別に解決する。
param を参照しない `input[4]` や `reg[5]` は従来どおりパース時に確定する(動作は変わらない)。

インスタンスの同一性は呼び出しキー全体(`callee` + `#(...)` + 引数列)で決まる。
`g#(W=4)(x)` と `g#(W=8)(x)` は別のインスタンスとして別ノード群を持ち、`g#(W=4)(x)` の 2 度目の呼び出しは同じインスタンスを共有する。

トレース(`-t`)と VCD では、インスタンスのノードは `#` を除いた名前で現れる(`g#(W=4)(x)` の出力ポート `y` なら `g(W=4)(x).y`)。
名前の `#` は観測対象外の内部ノードの印であるため、キーの形をそのままノード名に使わない。

`var[N]`(module 側)はジェネリック対象外で、従来どおり `param` / `#define` で即時解決する。

## 9. シミュレーション・セマンティクス

シミュレーションエンジンが 1 tick をどう進めるかの確定値を定義する。
実装の構造は [ARCHITECTURE.md §7](ARCHITECTURE.md) を参照。

### 9.1 ティック粒度

`#1` は 1 レッドストーンティックである。
ゲームティック素子や 1 レッドストーンティック未満のパルスは扱わない。

### 9.2 同一ティック内の評価順序

各 tick は 3 相で進む。

1. 順序素子の出力確定(リピータ、トーチ、コンパレータ、オブザーバの出力を `hist` の front から計算)
2. 組合せ網の不動点解決(ダストの値を MAX 合流で収束させる)
3. 順序素子の入力サンプリング(次 tick へ向けて `hist` を更新)

合流は max をとる単調更新なので、不動点は決定的に収束し、走査順に結果が依存しない。
逐次キュー方式と同じ結果を、順序非依存で保証する実装である。

### 9.3 信号伝搬と減衰

減衰は tick ごとに動的計算する。
ダスト 1 個につき −1、合流点は最大値をとる。

### 9.4 素子の更新タイミング

各素子の出力規則を一覧する。

- ダスト:即時(同一 tick で出力が決まる)
- 0tick リピータ `r0`:`out(T) = in(T) > 0 ? 15 : 0`(同一 tick、組合せ評価、§4.3)
- リピータ `rn`:`out(T) = in(T−n) > 0 ? 15 : 0`(1 tick パルスも保持)
- ロック可能リピータ:横入力が `> 0` の間は `out(T) = out(T−1)`(凍結)に退化
- トーチ:`out(T) = in(T−1) > 0 ? 0 : 15`
- コンパレータ:`out(T) = f(back(T−1), side(T−1))`
- コンパレータ(サイド未接続):`out(T) = back(T−1)`(パススルー)
- オブザーバ:`out(T) = in(T−2) != in(T−1) ? 15 : 0`(隣接 2 サンプルの変化検出)
- オブザーバ(エッジ亜種):`op` / `on` / `oe` は判定式だけを立ち上がり / 立ち下がり / 2値エッジに差し替える(§4.6.1)

### 9.5 入力変更の反映

`a = 10;` は次のティックの先頭で回路に反映される(Verilog のノンブロッキング相当)。
変数は `logic(引数)` で束縛したものだけが回路に影響する。

パルス代入 `a = v ~ w;` は、`a` に `v` を代入したうえで「`w` tick 後に `a = 0`」を予約する。
減算は各 tick 末(出力反映後)に行うので、`v` は入力反映時から数えてちょうど `w` tick 適用されてから 0 に戻る。
`w < 1` はエラーとなる。

期限前に `a` を通常代入すると予約は解除される。
パルス代入で上書きすれば新しい幅で置き換わる。

### 9.6 `#init` の定常判定

`#init` は全ノード値が前 tick と不変、かつ全順序素子のパイプラインが一様(`hist` が全段同値)になったら終了する。
`INIT_TIMEOUT` tick を超えるとエラーとなる(発振の検出)。
発振回路は `wait(n)` で抜けられる。

`#until(cond)` も同じ `INIT_TIMEOUT` を流用する。
`cond` が真になるまで tick を進め、超過でエラーとなる。

### 9.7 観測タイミング

`monitor` と出力変数は、ティック処理完了後の値を見る。

### 9.8 合流規則

同一 reg への複数チェーン合流は、すべて最大値をとる(ゲーム仕様準拠)。

入力ポートも同じ規則に従う。
var 駆動(`logic(引数)` の束縛)と回路内配線(チェーンやネスト呼び出しの直結)が同じ入力ポートへ給電する場合、値はすべての駆動の max になる。
ポートもレバーも配線も「同じ点への給電」であり、この規則はトップレベル呼び出しと階層インスタンスで共通である。

## 10. エラーと警告

継続不能はエラーとして停止し、続行できる注意は警告として stderr に出す。
警告は実行を止めない。

### 10.1 警告(続行する)

| 条件 | メッセージの趣旨 |
|---|---|
| 範囲外の var を回路 input に渡した | 0 か 15 に丸めて警告(変数ごとに 1 回) |
| wire や reg 素子の再代入 | 最終代入を採用して警告 |
| トーチの焼き切れ | 監視窓内のトグル超過。一定期間強制 OFF |

### 10.2 エラー(停止する)

| 条件 | 趣旨 |
|---|---|
| `const reg` の範囲外強度、再代入 | 値が不変の前提を破る |
| 未接続の output ポート | インスタンス化時に検出 |
| 名前の重複、未宣言変数 | 静的に検出 |
| 引数個数の不一致 | logic インスタンス化で検出 |
| `c` 単体、`r5` 以上 | 素子表記の妥当性 |
| 名前が素子名と衝突 | `r` / `cd` / `td` 等(§2.2) |
| リピーターやコンパレータ reg の後置代入 | `reg m; m = r;`(宣言時初期化に限る、§3.4) |
| `.side` を非対応 reg や始端に付けた | 横は入力専用で、対応素子は cmp / r* のみ |
| `#init` が `INIT_TIMEOUT` 超過 | 発振、永遠に成立しない条件 |
| `#until(cond)` の超過 | 同上 |
| `scan` の EOF や非数値 | 入力フォーマットの妥当性 |

### 10.3 診断表示(キャレット)

エラーや警告は Rust 風のキャレット表示で stderr に出す。
書式は `--> file:line:col` の後にソース行と `^` 下線が続く。

- 構文(字句や構文解析)エラー:正確な列を指す
- エラボレーションエラー:行レベル(行内容を下線)を指す

詳細な実装は [ARCHITECTURE.md §3.6](ARCHITECTURE.md) を参照。

### 10.4 デザインルールチェック(lint)

エラボレーションの後に **lint パス** が走り、動きはするが疑わしい構造を警告する(issue #48)。
実行時の警告(§10.1)と混ざらないよう、専用種別 `[lint]` にルール名を付けて stderr へ出す。
lint 警告は実行を止めない。

```
[lint] floating-reg: reg 'orphan' in logic 'sloppy' is not connected to anything
  --> examples/lint_demo.rv:18:5
```

| ルール名 | 条件 |
|---|---|
| `floating-reg` | reg がどこからも駆動されず、どこも駆動しない(完全に孤立) |
| `unused-wire` | wire が宣言されたがどのチェーンにも使われない |
| `unused-input` | input ポートが logic 本体で一度も使われない |
| `always-on-torch` | トーチの後ろ入力がどう操作しても 0 のまま(出力が常時 ON) |
| `unreachable-output` | 出力ポートがどう操作しても 0 のまま(例:ダスト 15 個の減衰死) |

前 3 つの静的ルールは宣言と結線の形だけを見る。
後 2 つは **上界解析** で判定する:各点が取りうる強度の上界を求め、上界 0 の点だけを報告する。
上界は過大評価はあっても過小評価はないので、報告された点が実行中に点灯することはない(誤検出なし)。
判定の仕組みは [ARCHITECTURE.md](ARCHITECTURE.md) を参照。

同じ logic を複数回インスタンス化しても、宣言由来の静的ルールは logic 名につき 1 回だけ出る。
上界解析のルールはトップレベルのインスタンス化ごとに、その素子・ポートを対象に出る。

CLI フラグ `-W error` を付けると、完走後に警告(§10.1 の実行時警告と lint の両方)が 1 件でもあれば終了コードが 1 になる。
CI で「警告ゼロ」を強制する用途を想定している。
`--json` モード(§7.4.1)では lint は `{"kind":"lint","rule":"...","line":N,"msg":"..."}` の JSONL になる。

5 ルールすべてを発火させるサンプルが `examples/lint_demo.rv` にある。

## 11. ゲームとの差異

本処理系の挙動は、本家のゲームのレッドストーン仕様と意図的に異なる箇所がある。
理由は単純化、決定性の確保、テキスト記述のしやすさのいずれかである。
ゲームを正確に模倣する処理系ではない点に注意する。

### 11.1 ダストの形状を持たない

ゲームのダストは配置で「点 / 十字」など形状が変わり、周囲への信号の伝え方が異なる。
本処理系では信号の行き先は reg のつなぎ方(チェーンの分岐)で決まるため、形状の区別を持たない。
ダストは `d` 一種類だけで、旧 `dx`(十字)記法は廃止した(issue #66)。

### 11.2 接続が有向である

実物のダストは双方向に信号を流すが、本処理系の接続は有向で、源から行き先へ一方向に流れる。
これは「テキスト上で `-` の左右に源と行き先を書く」記述モデルの自然な帰結である。
fan-in や fan-out(§6.4)はこの有向モデル上で「複数点を 1 点へ MAX 合流」「1 点を複数点へ複製」する糖衣として書ける。

### 11.3 ロック条件が緩い

ゲームのリピーターロックは「横の入力がリピーター / コンパレータからのとき」だけ作動する。
本処理系は横入力が `> 0` ならどんな経路(ダスト等)でもロックする。
源の素子種別を見ない単純なルールで、テキスト記述のしやすさを優先している。

### 11.4 0tick リピータを持つ

ゲームに真の 0tick リピータはなく、遅延 1 が最小である。
本処理系では「より高性能な回路をテキストで組む」ための拡張として `r0` を用意している(issue #37)。
0tick のトーチやコンパレータは反転や減算が非単調で、組合せループ検出を伴うエンジン拡張が要るため、別 issue で扱う。

### 11.5 オブザーバが入力信号の変化を見る

ゲームのオブザーバはブロック状態の更新で発火し、向きや QC(疑似配線)、出力面などの空間的性質を持つ。
本処理系は素子レベルで入力信号の変化のみを見る(向きや QC は扱わない)。

またゲームのオブザーバは変化全部を検出する 1 種類だけで、エッジ判定の亜種は持たない。
`op` / `on` / `oe`(§4.6.1)は、パルス整形回路をテキストで簡潔に書くための本処理系の拡張である(issue #58)。

### 11.6 ブロックを持たない

ゲームのブロックは、弱い動力と強い動力を区別しながら信号を通す伝導体で、トーチの取り付け台や壁越しの伝達といった空間的な役割を担う。
本処理系は「任意の 2 点を素子列で結ぶ」抽象で空間性を持たないため、これらの役割は接続を書くことそのもので代替される。
かつては 2 値化素子 `b`(入力 `> 0` なら 15)を提供していたが、同じ組合せ関数を 0tick リピータ `r0`(§4.3)が担うため、ブロック素子は廃止した(issue #75)。
