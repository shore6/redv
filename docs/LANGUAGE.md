# redv 言語仕様

`.rv` ファイルの文法と、シミュレーションエンジンの挙動(セマンティクス)を定義します。
概要・ビルド方法は [../README.md](../README.md) を参照してください。

---

## 1. 全体構造

- `logic name(input a, output y) { ... }` … レッドストーン回路の定義
- `module name() { ... }` … テストベンチ。`var` 宣言と `sim { ... }` ブロックを書く
- 文末は `;`。ブロックは `{}`。コメントは `//`(行)と `/* */`(範囲)
- ファイル内の全 `module` が **宣言順** に実行される(回路はモジュールごとに独立)

## 2. 宣言子

| 宣言 | 意味 |
|---|---|
| `input` / `output` | ポート。無条件で `reg` 型(= 点) |
| `wire` | 配線。素子列を代入する |
| `reg` | 点。単体素子を代入する |
| `const reg r = 15b;` | 素子も値も不変。**信号強度の指定が必須**(0–15、範囲外はエラー) |
| `mutable reg r = d;` | 素子は不変、値は回路によって変化 |
| `var` | テスト用の整数変数(C の `int` 相当)。**回路には組み込めない** |

## 3. 素子

| 表記 | 素子 | 動作 |
|---|---|---|
| `d` / `dx` | ダスト(点 / 十字) | 即時伝搬、1 個につき強度 −1 |
| `dn`(例 `d3`) | ダスト n 個 | `ddd...` と同じ |
| `r`, `r1`–`r4` | リピータ | n tick 遅延、出力は 15 に再増幅。1tick パルスも保持 |
| `t` | トーチ | 1 tick 遅延で反転(入力 > 0 → 0、入力 = 0 → 15) |
| `cc` / `cd` | コンパレータ(比較 / 減算) | 1 tick 遅延で **強度をそのまま** 通す(`c` 単体はエラー) |
| `b` | ブロック | 即時、強度は 0 か 15 のみ |

> `r` は `r1` と同じ(遅延 1)。`r5` 以上はエラー。`c` 単体はモード未指定のためエラー(`cc` / `cd` を使う)。

## 4. ワイヤーと点

```rv
wire w;
w = a-ddr2brdccbr4d3-b;   // a と b を素子列でつなぐ(チャンクは連結して書ける)
w = a-d-r2-t-b;           // '-' で区切って書いてもよい
w = a-b;                  // 長さ0のワイヤー(直結)
```

- ワイヤーの先頭と末尾は **宣言済みの reg / ポート**、中間は **素子のみ**
- 1 つの reg に複数のワイヤーが合流した場合、**強度の最大値** をとる(ゲーム仕様準拠)
- ワイヤーの再代入は **警告** を出し、最後の代入が採用される

## 5. sim ブロック

```rv
module test() {
    var a, y;
    sim {
        a = 0;
        y = not_gate(a);   // 初回で回路をインスタンス化、変数を入出力に束縛
        #init              // 定常状態まで待つ($time = 0 となる)
        a = 10;
        #1                 // 1 tick 進める
        ?monitor("時刻=%t | a=% | y=%\n", $time, a, y);
    }
}
```

- `v = logic名(変数...)` … 同じ「logic 名 + 引数列」は同一インスタンス。入力変数は継続的に束縛され、
  出力変数は毎 tick 更新される
- `#n` … n tick 進める。`#init` … 更新キューが空(定常状態)になるまで待つ
- `?monitor(...)` … Verilog の `$monitor` 風。sim 開始時に登録され、**すべてのウェイト
  (`#init` / `#n`)完了直後** に自動実行される(sim 内のどこに書いても全時刻が出力される)
- `monitor(fmt, ...)` … その場で出力。`%` / `%d` / `%t` が次の引数を消費、`%2d` のような幅指定可、
  `%%` で `%`
- `wait(n)` … `$time` を進めず、monitor も発火させずに n tick 待つ(発振回路用)
- `v = scan()` … **stdin** から空白/改行区切りの整数を 1 つ読み、変数 `v` に代入する。
  EOF(入力切れ)・非数値は **エラー**。回路へ束縛した変数は読み取り後も 0–15 にクランプされる。
  テストでは `redv foo.rv < input.txt` のように入力を固定すれば決定的に再現できる
- `$time` … `#init` 完了時(未使用なら sim 開始時)を 0 とした経過 tick
- `if` / `while` / `for` … C 準拠(`for (i = 0; i < 4; i = i + 1) { ... }`)

## 6. ディレクティブ

```rv
#define INIT_TIMEOUT 200      // #init のタイムアウト(既定 1000 tick)
#define BURNOUT_LIMIT 8       // トーチ焼き切れ: 窓内トグル回数(既定 8)
#define BURNOUT_WINDOW 30     // 同: 監視窓(既定 30 tick)
#define BURNOUT_COOLDOWN 30   // 同: 強制 OFF 期間(既定 30 tick)
#define MODE element          // 素子レベルモード(logic モードは将来対応)
#include "other.rv"           // ファイル取り込み(stdlogic は未実装で警告)
```

## 7. シミュレーション・セマンティクス(設計判断の確定値)

1. **ティック粒度** — `#1` = 1 レッドストーンティック。ゲームティック素子・1rstick 未満のパルスは
   扱わない。
2. **同一ティック内の評価順序** — 各 tick で「順序素子の出力確定 → 組合せ網(ダスト / ブロック)の
   不動点解決 → 順序素子の入力サンプリング」の 3 相。合流は max をとる単調更新なので不動点は決定的に
   収束し、走査順に結果が依存しない(逐次キュー方式と同じ結果を、順序非依存で保証する実装)。
3. **信号伝搬** — 減衰はティックごとに動的計算。ダスト 1 個につき −1、合流点は最大値。
4. **素子の更新タイミング**
   - ダスト / ブロック: 即時
   - リピータ `rn`: `out(T) = in(T−n) > 0 ? 15 : 0`(1tick パルス保持)
   - トーチ: `out(T) = in(T−1) > 0 ? 0 : 15`
   - コンパレータ: `out(T) = in(T−1)`(強度パススルー)
5. **#init の定常判定** — 全ノード値が不変かつ全順序素子のパイプラインが一様になったら終了。
   `INIT_TIMEOUT` tick を超えるとエラー(発振回路向けに `wait(n)` を用意)。
6. **入力変更の反映** — `a = 10;` は次のティックの先頭で回路に反映(Verilog のノンブロッキング相当)。
   変数は `logic(引数)` で束縛したものだけが回路に影響する。
7. **観測タイミング** — `monitor` / 出力変数は **ティック処理完了後** の値を見る。
8. **同一 reg への複数 wire 合流** — ゲーム仕様通り最大値。
9. **エラー / 警告(厳しめ)**
   - 範囲外の var → input: 0/15 に丸めて **警告**(変数ごとに 1 回)
   - const reg の範囲外強度・再代入: **エラー**
   - 未接続の output ポート: インスタンス化時に **エラー**
   - wire / reg 素子の再代入: **警告**(最終代入を採用)
   - トーチ焼き切れ: 窓内トグル超過で **警告** + 一定期間強制 OFF
   - 名前重複・未宣言変数・引数個数不一致・`c` 単体・`r5` 以上: **エラー**

## 8. サンプル

|---|---|
| `examples/not_gate.rv` | トーチ 1 本の NOT |
| `examples/or_gate.rv` | リピータ 2 本 + ダスト合流の OR |

| `examples/scan_and.rv` | `scan()` で stdin から 2 値を読んで AND に通す |

## 9. 既知の制限・今後

- コンパレータのサイド入力(減算 / 比較の第 2 入力)は未対応。チェーン内では強度パススルーとして動作
- ワイヤーは記述方向への単方向伝搬(双方向ダストは未対応)
- `logic` 内から別の `logic` を呼ぶ階層化は未対応
- `#define MODE logic` + `stdlogic`(ロジックレベル記法)は未対応(警告して無視)
- ゲームティック粒度・0tick パルス・準接続(QC)などの実機挙動は対象外。Minecraft JE の難読化解除後、
  実装準拠エンジンへの差し替えを想定
