# redv スタイルガイド

`.rv` を **読める形で書く** ための命名・スタイル・階層化の推奨をまとめる。
拘束的な規則ではなく、共通の慣行として揃えるレベルに留める。

- 言語仕様(何が書けるか)は [LANGUAGE.md](LANGUAGE.md) が一次情報。
  本書はそれを前提に **「どう書くと読みやすいか」** を扱う。
- 概要は [../README.md](../README.md)、内部設計は [ARCHITECTURE.md](ARCHITECTURE.md)。

## 1. このドキュメントの位置づけ

言語仕様どおりに書けば `.rv` は動くが、動くコードと読めるコードは別物である。
特に少し複雑な回路になると、次の 3 点で急に読みにくくなる。

- **中間変数の意味が薄れる。** `p` / `q` / `r1` / `r2` のような数式変数を並べると、
  読み手は式を追わないと役割が分からない。
- **`logic` の名前がぶれる。** 同じような名前の `logic` が増えると、書くときも読むときも
  どれがどのゲートか取り違える。名前の付け方は §2.4 で扱う。
- **遅延段数が追えなくなる。** 何 tick 後に信号が届くのかがコードから読み取れず、
  テストベンチの `#3` / `#6` の数字が根拠不明になる。
- **階層が見えなくなる。** チェーン文だけで最後まで書き切ろうとすると、
  1 logic の中に役割の違う素子列が混ざり、どこがどのゲートか分からなくなる。

対処は次の 3 方向で、§2〜§4 に対応する。

- 命名を整える(§2)
- 場面に合ったスタイルを選ぶ(§3)
- 階層に切って積む(§4)

## 2. 命名

### 2.1 素子名との衝突を避ける

`reg` / `wire` / ポートの名前は素子名と衝突できない
([LANGUAGE.md §2.2](LANGUAGE.md))。
チェーン内で「点」と「素子列」が区別できなくなるためで、
守らないと構文エラーか意図しない解釈になる。

使えない名前の代表:

| 種別 | 例 |
|---|---|
| 単体素子 | `d` / `o` / `r` / `t` |
| コンパレータ | `cc` / `cd` |
| オブザーバのエッジ亜種 | `op` / `on` / `oe` |
| 遅延つき素子 | `r2` / `d3` / `t0` のような素子 + 数字 |
| 素子列と読める連結 | `td`(トーチ + ダスト) / `rd` など |

単体では `a` / `b` / `c` / `x` / `s` は素子名でないので使ってよい。
`b` と `tb` はかつて素子名だったが、ブロック素子の廃止(issue #75)で識別子に使えるようになった。
`q` も文法上は使えるが「意味の薄い数式変数」になりがちなので、
役割を載せた名前(§2.3)に置き換えると良い。

### 2.2 入出力ポートの慣用

以下に寄せておくと、複数の logic を組み合わせるとき読み口が揃う。

| 形 | 推奨 |
|---|---|
| 1 入力 1 出力 | `input x, output y` |
| 2 入力 1 出力 | `input x1, input x2, output y` |
| バス版 | `input[N] x1, input[N] x2, output[N] y` |
| 加算器系 | `input a, input b, output sum, output carry` |
| 制御信号 | `input[N] select, output flag`(意味を優先) |

一般的な入出力ポートの変数名としては、`x1/x2/.../y` 以外に `in1/in2/.../out`などもよい。
ただし、その logic の役割にあった名称が望ましい。
対称なゲート入力は `x1, x2`、加算器のようにオペランドとして読ませたい対は `a, b` と使い分ける。
`a, b` の対は、ブロック素子 `b` の廃止(issue #75)で使えるようになった(§2.1)。

避けたい対は `a, b2` のような非対称な組である。

### 2.3 中間 reg / wire の慣用

よく出るパターンには短い慣用名がある。
新規に書くときはここに寄せると、他ファイルとの読み口が揃う。

| 用途 | 慣用名 | 例 |
|---|---|---|
| NOT の中間 | `na` / `nb` / `nx1` / `nx2` | `na = s_not(x1);` |
| 合流点(OR / NOR / AND の合流) | `s` | `s = d; x1 - r - s; x2 - r - s;` |
| コンパレータ | `cmp` | `reg cmp = cd;` |
| リピータロック reg | `m` | `reg m = r;` |
| ロック源 | `lk` | `reg lk;` |

バスの合流点は plain の `reg[W] s;` をそのまま使う(バス reg への素子代入はコンパレータ / リピーターのみで、`= d` は書けない。LANGUAGE.md §6.5)。
合流の挙動はスカラと同じ MAX 合流で、ダスト素子の有無で観測値は変わらない(stdlogic のゲートもこの形)。

避けたいのは `p` / `q` / `r1` / `r2` / `tmp` / `t1` のような役割が見えない一時変数。
XOR を書くなら中間は `sor` / `sand` / `nsand` のように **役割を名前に載せる**
(`examples/half_adder.rv` 参照)。

### 2.4 logic 名の慣用

logic 名の付け方には、業界に大きく二つの流儀がある。

- **標準セル流(回路図寄り)** — 機能を大文字略記 + 入力数で書く。
  `AND2` / `NAND2` / `INV` / `XNOR3` / `MUX2` のように。ASIC のセルライブラリの命名で、
  末尾の数字は **入力数(fan-in)**、`X1` / `X2` は駆動能力(サイズ)を表す。
- **RTL 流(Verilog モジュール)** — 機能ブロックは小文字 snake_case で意味を綴る。
  `full_adder` / `program_counter` / `alu` のように。信号は小文字、定数・パラメータは大文字、
  と層で住み分ける。

redv は素子〜ゲートレベルを直接書く言語なので、**大文字寄り** を推奨する。
reg / wire / 信号を小文字に寄せている(§2.2・§2.3)ぶん、logic 名を大文字にすると
呼び出しが視覚的に浮いて「ここはゲート/ブロック」と読めるからである。

```rv
sor = OR2(x1, x2);      // 大文字の OR2 が小文字の信号から浮く
```

推奨する形は 2 層に分ける。

| 層 | 命名 | 例 |
|---|---|---|
| 基本ゲート | 機能(大文字)+ 入力数 | `NOT` / `AND2` / `OR2` / `XOR2` / `NAND2` |
| 機能ブロック | 大文字 + `_`(多語) | `HALF_ADDER` / `FULL_ADDER` / `ADDER4` / `ALU` |

末尾の数字は **入力数(fan-in)** であって、バス幅ではない。`AND2` は 2 入力 AND を指す。
入力数はそのままにビット幅だけ変えたいとき(4 ビットバスのビット単位 AND など)、名前に幅を
埋めて `AND4` と書くと「4 入力」と誤読される。幅はジェネリック幅 `#(W=4)`
(`examples/generic_logic_width.rv`)で `AND2 #(W=4)` のように分離すると読み違えない。
入力が 1 本のゲート(`NOT` / `INV`)は数字を省く。

バンドルの stdlogic は例外で、小文字 + `s_` 接頭辞(`s_not` / `s_and` …)を使う。
これは「標準ライブラリの名前空間」を示す印で、自作の大文字ゲートと層が見た目で分かれる
ようにするためである(§4.1)。`#include "stdlogic"` したら `s_and(...)` で呼び、
自分でゲートを定義するときは上の大文字流に寄せると良い。

`examples/*.rv` の基本ゲート/ブロックはこの大文字流に揃えてある(`NOT` / `AND2` / `OR2` /
`XOR2` / `HALF_ADDER` など)。一方、各サンプル固有の一回きりの回路(`osc` / `split` / `via_rep`
など)は、役割を表す小文字名でよい。ゲート/ブロックとして再利用する意図のものを大文字に寄せる、
という使い分けである。


### 2.5 定数の使い分け

`param` と数値 `#define` は同じ定数表に入り、機能としては等価である([LANGUAGE.md §8.3](LANGUAGE.md))。
違いは構文が伝える意図だけなので、次の基準で使い分けると読み口が揃う。

| 用途 | 推奨 |
|---|---|
| 回路の設計パラメータ(バス幅・段数・繰り返し回数など) | `param` |
| 処理系の組み込み設定の上書き(`INIT_TIMEOUT` / `BURNOUT_*` / `MODE`) | `#define` |

`param NAME = 式;` は言語のトップレベル文で、この回路が持つ定数だという意図が構文に出る。
設計パラメータはこちらに寄せる。

`#define` はディレクティブで、既定値を持つ組み込み設定名の上書きが本来の役割である(issue #84)。
組み込み設定以外の定数をここに書くと、処理系の設定を変えているように読めてしまう。

### 2.6 命名で救えないもの

`na, nb, sor, sand` まで揃えても、素子が 20 個並ぶ logic は読めない。
そこは次の §3(スタイルの選択)と §4(階層化)で解く。

## 3. 回路スタイルの選択

`.rv` の回路記述は大きく 2 系統ある。

- **チェーン記法** `x1 - t - s;` — 素子列を線で書く。「地面に素子を並べる」感覚。
- **階層合成** `s = AND2(x1, x2);` — logic を関数のように呼び合わせる。「関数合成」感覚。

どちらか一方に統一する必要はなく、場面で使い分ける。

### 3.1 チェーン記法が向く場面

- **1〜2 段の合流やトーチ配線。** `AND2` のように 3 素子で終わる回路は
  チェーンが最も短く読める。
- **バスの一括配線。** `x1 - t - nx1;` のように N レーンを 1 文で書ける
  (`examples/bus_and4.rv`)。階層合成では出せない密度で置ける。
- **合流(MAX)を明示したいとき。** `x1 - r - s; x2 - r - s;` のように
  同じ点に別経路を落とす操作はチェーンでしか書けない。

例(`examples/and_gate.rv`):

```rv
logic AND2(input x1, input x2, output y) {
    reg s;
    s = d;
    x1 - t - s;    // NOT x1 を合流点へ
    x2 - t - s;    // NOT x2 を合流点へ(s は OR)
    s - t - y;     // y = NOT s
}
```

### 3.2 階層合成が向く場面

- **3 段以上の深いゲート。** チェーンで書き切ると素子列が長くなり、
  「どこがどのゲートか」が読めなくなる。ゲート単位で切り出すと役割が明示される。
- **同じサブ回路を複数箇所で使うとき。** logic 化しておけば挙動を一括で差し替えられる。
- **stdlogic を使うとき。** `#include "stdlogic"` で NOT / AND / OR / XOR / NAND / NOR / XNOR
  とリダクション(`s_rand` / `s_rnand` / `s_ror` / `s_rnor`)が使えるので、これらの合成で書くと
  自作ゲートを毎回定義せずに済む。多入力の AND / OR は arity 別の自作ゲートではなく、
  幅 `#(W=…)` を変えたリダクション 1 つで書く(LANGUAGE.md §8.2.1)。

例(悪い書き方 → 良い書き方):

```rv
// (悪) 中間変数が数式変数で役割が読めない
logic XOR2(input x1, input x2, output y) {
    reg p, q, r1, r2;
    // ... 素子列がずらずら
}

// (良) 名前に役割を載せて階層で組む(logic 名は §2.4 の大文字流)
logic XOR2(input x1, input x2, output y) {
    reg sor, sand, nsand;
    sor   = OR2(x1, x2);
    sand  = AND2(x1, x2);
    nsand = NOT(sand);
    y     = AND2(sor, nsand);
}
```

### 3.3 混在させる指針

同じ logic の中でチェーンと階層合成を混ぜてよいが、次の順序に並べると読みやすい。

1. 内部 `reg` の宣言
2. 合流点の初期化(`s = d;` など)
3. チェーン文(素子レベルの結線)
4. 階層合成(logic 呼び出し)
5. 出力の駆動(`y = ...`)

「素子で作る点」と「合成で作る点」を分けて置くと、
どこまでが素子レベル、どこからがブロックレベルかが視覚的に分かる。

### 3.4 スタイル選択の目安

| 目安 | 推奨スタイル |
|---|---|
| 素子 3 個以下の 1 段ゲート | チェーン |
| 3 段以上、または複数ゲートの合成 | 階層合成(できれば stdlogic) |
| バス演算 | チェーン(N レーンを 1 文で) |
| CPU / ALU のようなブロック合成 | 階層合成 |

## 4. 階層設計

大きな回路(CPU など)を組むときの目安をまとめる。
現状 redv の中〜大規模実装例は `curcuit/td4_cpu/` が代表例(実装中)。
本節は材料が揃うにつれて追記する予定で、初版では最小限に留める。

### 4.1 stdlogic を土台にする

`#include "stdlogic"` を最下層に置き、その上に

- 中間ブロック(半加算器・全加算器・4 ビット加算器 など)
- 大ブロック(ALU・レジスタ・CPU 全体 など)

を **ボトムアップに** 積むと、各層を独立にテストできる。
stdlogic のゲートは小文字 + `s_` 接頭辞(`s_not` / `s_and`)、自作のゲート/ブロックは
大文字(§2.4)と、名前の見た目で層が分かれるので、ライブラリと自作の取り違えが起きにくい。

### 4.2 遅延段数コメント

各 logic のヘッダに **伝搬遅延**(入力の変化から出力が確定するまでの tick 数)を
コメントで書いておくと、テストベンチの `#N` や上位ブロックの遅延見積りの根拠になる。
stdlogic なら `src/stdlib/stdlogic.rv` 冒頭のコメント(各ゲートの遅延一覧)がそのまま使える。

書き方は二つの形を使い分ける。どちらの形にも固定タグ `delay:` の行を入れる。
タグを固定しておくと、`grep` でプロジェクト全体の遅延一覧が引けるからである。
logic 名はコメントに書かない。直下の宣言と重複するうえ、
リネーム時にコメントだけ古い名前が残るためである。

1 出力の小物ゲートは `//` 2 行で書く。1 行目に遅延タグ、2 行目に説明を置く。

```rv
// delay: 2
// 2 入力 AND
logic AND2(input x1, input x2, output y) { ... }
```

多出力のゲートや機能ブロックは `/* */` で書き、説明のあとに出力ごとの遅延行もしくは最大の伝搬遅延を記述する。


```rv
/* 半加算器 sum = x1 XOR x2、carry = x1 AND x2。
 * delay:
 *   sum   : 4
 *   carry : 2
 */
logic HALF_ADDER(input x1, input x2, output sum, output carry) { ... }
```

運用の注意:

- 推奨する遅延は伝搬遅延であって、**テストベンチに書く待ち数ではない**。
  sim の代入は次 tick に反映されるため、遅延 N tick の出力を読むには `#(N+1)` 待つ
  (`examples/and_gate.rv` の `#3  // トーチ2段 = 遅延2tick + 入力反映1tick`)。
  正確な数を数えたくないときは `#init` / `#until` で待てばよい。
- 伝搬遅延は加算で合成できる。上位ブロックの遅延は経路上の logic の遅延の和
  (経路が複数あれば最大値)なので、括弧の経由メモ(`XOR2 経由` など)が
  再計算の根拠になる。
- 入力や状態で遅延が変わるブロック(桁上げ連鎖など)は、最悪値に条件を添えて書く
  (例: `遅延: 8 (最悪、桁上げ全段伝搬)`)。
- 小物ゲートを束ねたライブラリファイルでは、logic ごとのヘッダの代わりに
  ファイル冒頭の一覧表へまとめてもよい(`src/stdlib/stdlogic.rv` の流儀)。

### 4.3 ファイル分割の目安

- **単発モジュール(サンプル、テストベンチ)** — 1 ファイルに全部書く
  (`examples/*.rv` の流儀)。
- **中規模プロジェクト(CPU など)** — 論理ブロック単位でファイルを分ける
  (`ALU.rv` / `REGISTER.rv` / `TOP.rv` のように)。
  `#include` で組み合わせる。

分割の粒度は「1 ファイル 1 責務」を目安にする。
テストベンチ側は最上位に 1 個の `module` を置き、そこから下位 logic をインスタンス化する。
